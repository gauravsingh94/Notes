System Design,
simply explained.

Everything you need to walk into a system design round at Google, Amazon, Meta, Microsoft, or any MNC — every concept, every pattern, every classic problem — written in plain English with real-world examples.

What is System Design?

System design is the process of defining the architecture, components, modules, interfaces, and data flow of a software system to satisfy a set of requirements. In simple words — it's the blueprint of how a product like Instagram or YouTube is built behind the scenes.

When you open YouTube and a video starts playing in 200 milliseconds — that smoothness is not magic. It's the result of dozens of carefully designed pieces: load balancers, caches, CDNs, databases spread across the world, message queues, and more. System design is the study of why each piece exists and how they fit together.

Real-world analogy

Imagine you're opening a restaurant. You need to decide: how many tables (capacity), how many waiters (servers), where the kitchen is (database), how orders flow from waiter to chef (API contracts), what happens when 500 customers arrive at once (scaling), and what if a waiter quits (fault tolerance). System design is exactly this — but for software.

Why interviewers care

Coding rounds test if you can build a feature. System design rounds test if you can build a product. They reveal how you think about trade-offs, how you handle ambiguity, and whether you can communicate technical decisions clearly — exactly what senior engineers do every day.

HLD vs LLD — What's the difference?

Interviewers usually split system design into two flavors. Knowing which one you're being asked is half the battle.

Quick rule

If the problem starts with "Design X" and X is a product (Twitter, YouTube, Dropbox) → it's HLD. If X is an entity (Parking Lot, Elevator, Snake & Ladder) → it's LLD.

The Universal Interview Framework

Never start drawing immediately. Follow this 6-step framework — it works for any HLD question, in any company.

Step 1 — Clarify Requirements (5 min)

Ask questions to narrow down the scope. Split into Functional (what the system does) and Non-Functional (how well it does it).

  • Functional: "Should users be able to like a post? Comment? Repost?"
  • Non-Functional: "How many daily active users? What's the read:write ratio? Latency target?"

Step 2 — Capacity Estimation (5 min)

Do back-of-envelope math. Number of users × actions per day → reads/sec, writes/sec, storage, bandwidth. (Full guide in Chapter 04.)

Step 3 — API Design (5 min)

Sketch 3–4 endpoints. Show you think about contracts before code.

POST /tweets         { userId, text, mediaUrl }
GET  /feed/{userId}  → list of tweets
POST /follow         { followerId, followeeId }

Step 4 — High-Level Architecture (15 min)

Draw the boxes: Client → CDN → Load Balancer → App Servers → Cache → DB → Queue → Workers. This is the headline diagram.

Step 5 — Deep Dive (15 min)

Pick 1–2 components and zoom in. Database schema. Cache strategy. How feed is generated. This is where you score the most points.

Step 6 — Bottlenecks & Trade-offs (5 min)

Talk about what breaks at scale and what you'd do about it. Mention monitoring, alerts, single points of failure.

Remember

Interviewers don't expect a perfect system. They want to see structured thinking, clear communication, and awareness of trade-offs.

Worked example — Designing Twitter

Assumption: 300M daily active users, each posts 2 tweets/day, average tweet = 300 bytes.

Writes/sec: 300M × 2 / 86,400 ≈ 7,000 tweets per second.

Reads/sec: Assume each user opens feed 10×/day → 300M × 10 / 86,400 ≈ 35,000 reads/sec. Read:write ratio is 5:1 — heavily read-optimized.

Storage/year: 600M tweets/day × 300 bytes × 365 ≈ 65 TB/year of raw text. Add media → ~10× that.

Scalability

Scalability is the ability of a system to handle more load by adding resources. There are two flavors:

Vertical Scaling (Scale Up)

Add more power to a single machine — more CPU, more RAM, faster disk. Simple but limited (you eventually hit the biggest machine money can buy) and creates a single point of failure.

Horizontal Scaling (Scale Out)

Add more machines and distribute load across them. Theoretically unlimited but introduces complexity — coordination, networking, consistency.

VerticalHorizontal
CostCheap initially, expensive at topLinear cost growth
ComplexityLowHigh (load balancing, sync)
LimitHardware ceilingTheoretically unlimited
DowntimeYes, to upgradeNo, add servers live
Failure toleranceSingle point of failureBuilt-in redundancy
Real-world example

A traditional bank's mainframe was vertical scaling — one giant machine. Netflix, on the other hand, runs on thousands of small AWS servers — horizontal. When you scroll Netflix, hundreds of these machines together handle the request.

Latency vs Throughput

These two are often confused. They are not the same thing.

  • Latency = time taken for one request to complete. Measured in ms. Lower is better.
  • Throughput = number of requests completed per second. Measured in RPS (requests/sec). Higher is better.
Highway analogy

Imagine a highway. Latency is how long it takes one car to cross from start to end. Throughput is how many cars cross the toll booth per minute. A wider highway (more lanes) increases throughput but doesn't necessarily reduce latency. A faster speed limit reduces latency but doesn't change how many cars fit.

In interviews, always state your target — e.g., "P99 latency under 200ms, throughput of 10K RPS." P99 means 99% of requests are faster than this number.

Load Balancing

A load balancer sits between clients and your servers, distributing incoming traffic across multiple machines. It also detects when a server is down and stops sending requests to it.

┌──────────────┐ │ Clients │ └──────┬───────┘ │ ┌──────▼───────┐ │ Load Balancer│ └──┬───┬───┬───┘ │ │ │ ┌─────▼┐ ┌▼──┐ ┌▼──┐ │ S1 │ │S2 │ │S3 │ └──────┘ └───┘ └───┘

Layer 4 vs Layer 7 Load Balancing

  • L4 (Transport): Routes based on IP & port. Fast but blind to content. Example: AWS NLB.
  • L7 (Application): Reads HTTP headers, URL, cookies. Can route /api/users to one server and /api/videos to another. Slower but smarter. Example: NGINX, AWS ALB.

Algorithms

  1. Round Robin — rotate through servers in order. Simple, fair if all servers are equal.
  2. Weighted Round Robin — give powerful servers more requests.
  3. Least Connections — send next request to the server with fewest active connections. Great for long-lived connections.
  4. IP Hash — same client IP always lands on the same server. Used for session stickiness.
  5. Least Response Time — pick the fastest-responding server.
Example

When you visit amazon.in, DNS gives you the load balancer's IP. The LB picks (say, via least connections) one of 10,000 servers and forwards your request. If one server crashes mid-request, the LB notices via health checks and removes it from rotation within seconds.

Caching

A cache is a small, fast storage layer that holds recently-used data to avoid recomputing or refetching. Caches are the single biggest reason modern apps feel fast.

The cache rule

Memory is ~100× faster than disk. Disk is ~100× faster than network. Caching pulls data closer to the user along this hierarchy.

Where caches live

  • Browser cache — your browser stores images, CSS, JS.
  • CDN cache — edge servers worldwide (e.g., Cloudflare).
  • Application cache — in-memory in your app server.
  • Distributed cache — Redis, Memcached — shared across servers.
  • Database cache — query result cache inside the DB.

Caching Strategies

1. Cache-Aside (Lazy Loading)

App checks cache → if miss, reads DB → writes to cache → returns. Most common pattern.

def get_user(id):
    user = cache.get(id)
    if user is None:
        user = db.query("SELECT * FROM users WHERE id=?", id)
        cache.set(id, user, ttl=3600)
    return user

2. Read-Through

App talks only to cache. Cache itself fetches from DB on miss. Simpler app code.

3. Write-Through

Every write goes to cache and DB simultaneously. Cache always in sync, but writes are slower.

4. Write-Back (Write-Behind)

Writes go to cache first, async to DB later. Very fast writes but risk of data loss if cache crashes.

5. Write-Around

Writes go directly to DB, skipping cache. Good when written data is rarely read again.

Eviction Policies

Cache is small — it must throw things out. How does it choose?

  • LRU (Least Recently Used) — most common. Evict what hasn't been touched in a while.
  • LFU (Least Frequently Used) — evict what's been accessed least often.
  • FIFO — oldest in is oldest out.
  • TTL — evict after a fixed time.
Real example — Instagram feed

When you open Instagram, your feed isn't computed from scratch. It was pre-computed and stored in Redis. The first 50 posts load in 30ms because they come from cache — not from a database with 500M rows.

Content Delivery Network (CDN)

A CDN is a network of servers spread around the globe that store copies of your static content (images, videos, CSS, JS) close to the user.

If you're in Chennai and a website's main server is in Virginia, every image would take ~250ms to reach you. With a CDN, the same image is cached at a server in Mumbai or Chennai — it arrives in 10ms.

Two flavors

  • Pull CDN — fetches content from origin on first request, caches it. Cheaper, used by most websites.
  • Push CDN — you upload content directly to the CDN. Used for large videos / Netflix-style streaming.
Real example

When you watch a YouTube video, it isn't streamed from California — it's streamed from a Google edge server inside your ISP's network. That's why 4K loads instantly.

SQL vs NoSQL

ExamplesMySQL, PostgreSQL, OracleMongoDB, Cassandra, DynamoDB, Redis

ScalingVertical mostlyHorizontal natively
ConsistencyStrong (ACID)Often eventual (BASE)

NoSQL sub-types

  • Key-Value (Redis, DynamoDB) — like a giant hash map. Ultra-fast lookups.
  • Document (MongoDB) — stores JSON-like documents. Each row can have different fields.
  • Column-Family (Cassandra, HBase) — stores rows where each can have millions of columns. Great for time-series.
  • Graph (Neo4j) — nodes and edges. Best for social networks, fraud detection, recommendations.
When to pick what

Bank balance: SQL — must be ACID, no room for "eventually correct". Instagram likes count: NoSQL — being off by 2 likes for 1 second is fine. Friend recommendations: Graph DB — relationships are first-class.

Database Indexing

An index is like the back-of-book index — it lets the DB jump straight to the row without scanning the whole table.

Without an index, looking up WHERE email = 'a@b.com' in a 100M-row table means reading all 100M rows. With an index on email, the DB uses a B+ tree to find it in ~log(100M) ≈ 27 steps.

Types of indexes

  • Primary index — automatically on the primary key. Unique.
  • Secondary index — explicitly created on other columns.
  • Composite index — on multiple columns, e.g., (country, city). Order matters!
  • Unique index — also enforces uniqueness.
  • Full-text index — for searching text inside long strings.
Watch out

Indexes speed up reads but slow down writes (every insert must update the index). Don't index everything — only what you query.

Database Replication

Replication = keeping multiple copies of the same data on different machines. Why? For availability (one dies, others survive) and read scaling (multiple replicas handle reads).

Master-Slave (Primary-Replica)

One master accepts all writes. Slaves copy from master and serve reads. Most common setup.

Writes → ┌─────────┐ │ Master │ └────┬────┘ │ replication ┌────────┼────────┐ ▼ ▼ ▼ ┌─────┐ ┌─────┐ ┌─────┐ │ R1 │ │ R2 │ │ R3 │ ← Reads └─────┘ └─────┘ └─────┘

Master-Master (Multi-Primary)

Multiple nodes accept writes. Higher write capacity but conflict resolution is painful.

Sync vs Async replication

  • Synchronous — master waits for replicas to confirm before acknowledging. Safer, slower.
  • Asynchronous — master replies immediately, replicas catch up later. Faster, but if master dies you lose the last few seconds.

Sharding (Partitioning)

Sharding means splitting one big database into multiple smaller databases, each holding a subset of the data. Each shard is independent.

Sharding strategies

  1. Range-based — users A–F on shard 1, G–M on shard 2... Easy but creates hotspots.
  2. Hash-based — shard = hash(userId) % N. Even distribution but resharding when N changes is painful (see Consistent Hashing).
  3. Geographic — Asian users on Asian servers, EU on EU.
  4. Directory-based — a lookup service maps each key to its shard. Flexible but the directory itself is a bottleneck.
Sharding pain points

(1) Cross-shard joins are hard. (2) Rebalancing when traffic skews. (3) Unique IDs must be global. (4) Transactions across shards need distributed protocols.

CAP Theorem

The most famous theorem in distributed systems. It says that a distributed database can guarantee only 2 out of 3 of these properties:

  • C — Consistency: every read returns the latest write or an error.
  • A — Availability: every request gets a response (maybe stale).
  • P — Partition Tolerance: system keeps working even if the network splits.

In reality, network partitions will happen — so you must always choose P. The real choice is between C and A:

System typeChoiceExamples
CP — Sacrifice availability for consistencyRefuse to answer if unsureMongoDB, HBase, Redis (single)
AP — Sacrifice consistency for availabilityAlways answer, may be staleCassandra, DynamoDB, CouchDB
CA — Only possible without partitions(Theoretical for distributed)Traditional single-node SQL
Real choice

Bank transfer: CP — better to say "try again" than show wrong balance. Instagram feed: AP — better to show a slightly old feed than no feed at all.

ACID vs BASE

ACID (SQL world)

  • Atomicity — all or nothing. If transfer fails halfway, nothing happens.
  • Consistency — DB always moves from valid state to valid state.
  • Isolation — concurrent transactions don't see each other's half-baked work.
  • Durability — once committed, data survives crashes.

BASE (NoSQL world)

  • Basically Available — system always responds.
  • Soft state — state may change over time (even without writes).
  • Eventually consistent — given enough time, all replicas converge.

Consistency Models

  • Strong consistency — every read sees the latest write. Slow at scale.
  • Eventual consistency — replicas converge eventually. Used by DNS, Instagram likes.
  • Causal consistency — if A happens-before B, all nodes see A before B. (Used in chat apps so messages don't appear out of order.)
  • Read-your-writes — you always see what you just wrote, even if others don't yet.
  • Monotonic reads — once you've seen a value, you never see an older one.

Consistent Hashing

The classic problem: you have N servers and use server = hash(key) % N to route requests. Now you add or remove a server — suddenly almost every key remaps. Cache invalidated. Disaster.

Consistent hashing solves this elegantly. Imagine a circular ring (0 to 2³²). Hash each server onto the ring. Hash each key onto the ring. A key is owned by the next server clockwise on the ring.

Key K1 ↓ ┌───────●───────┐ │ │ │ Server A Server B │ │ │ └───────●───────┘ Key K2 → owned by B

When a server is added or removed, only ~1/N keys move — not all. This is how Cassandra, DynamoDB, and most distributed caches scale.

Virtual nodes: Each physical server is mapped to many points on the ring, giving smoother distribution.

Message Queues & Pub/Sub

A message queue decouples producers from consumers. Producer drops a message → queue holds it → consumer picks it up when ready.

Producer → [ Q1 ][ Q2 ][ Q3 ] → Consumer

Why use queues?

  • Decoupling — services don't need to know about each other.
  • Buffering — handles traffic spikes.
  • Async processing — don't make the user wait for slow tasks (email, video encoding).
  • Retries — if consumer fails, message stays in queue.

Queue vs Pub/Sub

  • Queue (point-to-point): one message → one consumer. Example: RabbitMQ, SQS.
  • Pub/Sub (broadcast): one message → all subscribers. Example: Kafka, Google Pub/Sub.

Kafka in plain English

Kafka is a distributed commit log. Producers append messages to topics. Topics are split into partitions for parallelism. Consumers read at their own pace using offsets. It can handle millions of messages per second and replay history.

Real example — Uber

When you book a ride, the request goes to a Kafka topic. Multiple services consume it: pricing, driver matching, ETA calculation, fraud detection — all in parallel, all decoupled. If fraud detection is slow, ride matching doesn't wait.

API Gateway

In a microservices world, the client doesn't talk to 50 different services. It talks to one entry point — the API Gateway. The gateway routes, authenticates, rate-limits, logs, and aggregates.

Responsibilities

  • Routing — /users → User Service, /orders → Order Service
  • Authentication & Authorization (JWT validation)
  • Rate limiting
  • Request/response transformation
  • Logging & metrics
  • SSL termination
  • Caching

Tools: Kong, AWS API Gateway, NGINX, Apigee.

Rate Limiting

Rate limiting protects your system from being overwhelmed — by abusive users, bots, or buggy clients.

Algorithms

1. Token Bucket

Bucket holds N tokens. Refilled at R tokens/sec. Each request consumes 1 token. No tokens → reject. Allows bursts up to bucket size. Used by AWS.

2. Leaky Bucket

Requests enter the bucket. Leak out at constant rate. If bucket overflows → reject. Smooths traffic.

3. Fixed Window

Count requests in each 1-minute window. Reset to 0 each minute. Simple but spiky at boundaries.

4. Sliding Window Log

Keep timestamp of each request. Count requests in the last 60s exactly. Accurate but memory-heavy.

5. Sliding Window Counter

Hybrid of fixed window and log. Most practical.

Where it's used

Twitter: 300 tweets per 3 hours per user. GitHub: 5,000 API calls per hour per token. Login endpoints: 5 attempts per minute per IP (to prevent brute force).

Forward Proxy vs Reverse Proxy

  • Forward Proxy sits in front of clients. The server doesn't know who the real client is. Used for privacy, geo-bypass, content filtering. (VPN-like.)
  • Reverse Proxy sits in front of servers. The client doesn't know which backend served them. Used for load balancing, caching, SSL termination. (NGINX, Cloudflare.)
Forward Proxy: Client → [Proxy] → Internet → Server Reverse Proxy: Client → Internet → [Proxy] → Server

Communication Protocols

REST

HTTP-based, stateless, uses verbs (GET, POST, PUT, DELETE). Easy to debug, universally supported. The default for public APIs.

GraphQL

Client specifies exactly what fields it wants. One endpoint serves any query. Great for mobile apps that want to avoid over-fetching.

query {
  user(id: 5) {
    name
    posts { title }
  }
}

gRPC

Google's RPC framework using HTTP/2 and Protocol Buffers (binary, fast). 5–10× faster than REST. Used for internal microservice-to-microservice calls.

WebSockets

Persistent, bidirectional channel between client and server. Both sides can push at any time. Used for chat, live dashboards, multiplayer games.

Server-Sent Events (SSE)

One-way from server to client over HTTP. Lightweight alternative to WebSockets for notifications.

Long Polling

Client sends request, server holds it until data arrives or timeout. Old-school but works everywhere.

Use caseBest pick
Public APIREST
Mobile app, varied data needsGraphQL
Internal service-to-servicegRPC
Real-time chat / gamingWebSockets
Notifications, stock tickersSSE

Bloom Filters

A Bloom filter is a probabilistic data structure that tells you whether an element is "definitely not in the set" or "possibly in the set". It uses very little memory.

How it works: An array of bits + several hash functions. To add an item, hash it with all functions and set those bits to 1. To check, hash again — if all bits are 1, it might be there; if any is 0, it's definitely not.

Where it shines

Medium / Quora: "Has this user already seen this article?" Check Bloom filter first — if no, skip the DB. Saves billions of DB queries.

Chrome: Stores known malicious URLs in a Bloom filter to check every site you visit instantly.

Distributed System Concepts

Leader Election

Among multiple nodes, one must be elected the leader to coordinate. Algorithms: Bully, Raft, Paxos, ZooKeeper-based. Used in Kafka, MongoDB replica sets, Kubernetes.

Heartbeats

Nodes periodically send "I'm alive" pings. Missing 3 heartbeats → declared dead. Triggers failover.

Gossip Protocol

Each node randomly tells a few neighbors what it knows. Information spreads exponentially. Used by Cassandra, Redis Cluster.

Quorum

For N replicas, require W writes and R reads such that W + R > N. Guarantees you'll read the latest write. Cassandra uses this.

Idempotency

An operation is idempotent if running it twice has the same effect as running it once. Critical for retries. PUT /user/5 {name:"A"} is idempotent; POST /payments usually isn't (which is why you need idempotency keys).

Two-Phase Commit (2PC)

Coordinator asks all nodes "can you commit?" → if all say yes, tells all to commit. Guarantees atomicity across nodes but blocks if coordinator fails.

OOP — The Four Pillars

1. Encapsulation

Bundle data and methods inside a class, and hide internal state from outside. Like a TV remote — you press buttons, you don't see the circuit.

class BankAccount {
    private double balance;          // hidden
    public void deposit(double x) {   // controlled access
        if (x > 0) balance += x;
    }
}

2. Inheritance

A class inherits properties from a parent. Reuse code, model "is-a" relationships.

class Animal { void breathe() {...} }
class Dog extends Animal { void bark() {...} }
// Dog "is-a" Animal

3. Polymorphism

One name, many forms. Two flavors:

  • Compile-time (overloading): same method name, different parameters.
  • Runtime (overriding): subclass redefines a parent method. Decided at runtime.
Animal a = new Dog();
a.makeSound();   // calls Dog's version, not Animal's

4. Abstraction

Show what a thing does, not how. Achieved via abstract classes and interfaces. Driving a car — you turn the wheel, you don't think about hydraulics.

SOLID Principles

Five principles that make OOP code clean and maintainable. Asked in almost every LLD interview.

S — Single Responsibility

A class should have one and only one reason to change. InvoicePrinter shouldn't also save to DB.

O — Open/Closed

Classes should be open for extension, closed for modification. Add new payment methods by creating new classes, not by editing existing ones with if/else.

L — Liskov Substitution

You should be able to replace a parent class with any of its subclasses without breaking behavior. If Square extends Rectangle but setWidth also changes height, it breaks code expecting a Rectangle.

I — Interface Segregation

Don't force classes to implement methods they don't need. Prefer many small interfaces over one fat one.

D — Dependency Inversion

Depend on abstractions, not on concrete classes. Your OrderService should depend on PaymentProcessor (interface), not on StripeProcessor directly.

Why SOLID matters

SOLID code is easier to test, easier to change, and easier to onboard a new engineer to. Every design pattern below is essentially a recipe for following SOLID.

UML Diagrams

UML (Unified Modeling Language) is a visual notation for object-oriented systems. The two diagrams you'll be asked to draw:

Class Diagram

Shows classes, their attributes, methods, and relationships:

  • Association — uses (line)
  • Aggregation — has-a, weak (hollow diamond). Department has Professors; if Department closes, Professors still exist.
  • Composition — owns, strong (filled diamond). House has Rooms; close the house, rooms vanish.
  • Inheritance — is-a (open arrow)
  • Realization — implements interface (dashed arrow)

Sequence Diagram

Shows the order of method calls between objects over time. Used to depict flows like "user places an order".

Creational Design Patterns

Patterns for object creation. They abstract the instantiation process.

1. Singleton

Ensure a class has only one instance. Use for config, logging, DB connection pool.

class Logger {
    private static Logger instance;
    private Logger() {}
    public static synchronized Logger getInstance() {
        if (instance == null) instance = new Logger();
        return instance;
    }
}

2. Factory Method

Define an interface for creating an object, let subclasses decide which class to instantiate.

interface Shape { void draw(); }
class ShapeFactory {
    Shape create(String type) {
        if (type.equals("circle"))  return new Circle();
        if (type.equals("square"))  return new Square();
        throw new IllegalArgumentException();
    }
}

3. Abstract Factory

Factory of factories. Produce families of related objects without specifying concrete classes. (E.g., a UIFactory that returns either Mac or Windows widgets.)

4. Builder

Construct complex objects step-by-step. Avoids constructors with 10 parameters.

Pizza p = new Pizza.Builder()
            .size("L")
            .cheese(true)
            .topping("mushroom")
            .build();

5. Prototype

Create new objects by copying an existing one (cloning). Useful when creation is expensive.

Structural Design Patterns

Patterns about composing classes and objects into larger structures.

1. Adapter

Make two incompatible interfaces work together — like a power-plug adapter.

// Existing: MediaPlayer plays mp3
// Need: play mp4
class Mp4Adapter implements MediaPlayer {
    Mp4Player player = new Mp4Player();
    public void play(String file) { player.playMp4(file); }
}

2. Decorator

Add new behavior to an object dynamically without changing its class. Java's I/O streams use this everywhere.

Coffee c = new Coffee();
c = new MilkDecorator(c);
c = new SugarDecorator(c);
c.cost();  // base + milk + sugar

3. Facade

Provide a simple interface to a complex subsystem. Like a hotel concierge — you don't talk to housekeeping, laundry, and the kitchen separately.

4. Composite

Treat individual objects and groups of objects uniformly. Used for tree structures — file system, UI components.

5. Proxy

A stand-in for another object — for lazy loading, access control, caching, logging.

6. Bridge

Decouple abstraction from implementation so both can vary independently. E.g., RemoteControl (abstraction) and Device (implementation) evolve separately.

7. Flyweight

Share common state across many objects to save memory. Used in text editors where each character object shares font/formatting data.

Behavioral Design Patterns

Patterns about communication between objects.

1. Observer

One-to-many dependency: when one object changes, all observers are notified. Used in event systems, pub/sub.

// YouTube channel notifies all subscribers when new video posted
channel.subscribe(user1);
channel.subscribe(user2);
channel.notifyAll(newVideo);

2. Strategy

Define a family of algorithms, encapsulate each, make them interchangeable at runtime.

PaymentStrategy s = isInternational
    ? new PayPalStrategy()
    : new UpiStrategy();
checkout.pay(s, 100);

3. Command

Encapsulate a request as an object. Enables undo, queuing, logging. Used in text editors (each keystroke is a command).

4. State

Allow an object to change behavior when its state changes. Like a traffic light cycling Red → Green → Yellow.

5. Template Method

Define the skeleton of an algorithm in a base class, let subclasses override specific steps. prepareDrink() has steps; Tea and Coffee override the brewing step.

6. Iterator

Access elements of a collection sequentially without exposing its internal structure. Java's Iterator<T>.

7. Chain of Responsibility

Pass a request along a chain of handlers; each decides to handle it or pass on. Used in middleware (auth → logging → cache → controller).

8. Mediator

Define an object that encapsulates how a set of objects interact. Air traffic control mediates between planes so they don't talk directly.

9. Memento

Capture an object's state so it can be restored later. Implements Ctrl+Z.

10. Visitor

Add new operations to a class hierarchy without modifying the classes. Used in compilers for AST traversal.

Design a URL Shortener (bit.ly)

Requirements

  • Shorten a long URL into a 7-character code.
  • Redirect short → long in <100ms.
  • Handle 100M new URLs/month, 10:1 read:write ratio.
  • Codes should be hard to guess.

Capacity

100M / 30 / 86400 ≈ 40 writes/sec; 400 reads/sec. Over 5 years → 6 billion URLs. Each row ~500 bytes → ~3 TB storage.

Short code generation — two approaches

  1. Hash-based: MD5(longUrl) → take first 7 chars (Base62 encoded). Handle collisions by appending a salt.
  2. Counter + Base62: Use a distributed counter (like a Snowflake ID). Convert decimal to Base62 → 7 chars give 62⁷ ≈ 3.5 trillion combinations.

Architecture

Client → CDN → LB → App Servers │ ├─→ Redis (short→long cache) └─→ Sharded DB (Cassandra/MySQL)

Reads

GET /abc1234 → app checks Redis first (hot URLs) → 90% cache hit → DB fallback. Use 301 (permanent) redirect; browsers cache it.

Trade-offs

Counter approach gives predictable IDs (security risk for guessing). Hash approach has collisions but is unpredictable. Use a hybrid in production.

Design Twitter / News Feed

The core question

How does Twitter show you a feed of tweets from people you follow, instantly, when each user follows thousands?

Two approaches

1. Pull model (Fan-out on read)

When user opens feed, query all followees' tweets, merge, sort by time. Cheap writes, expensive reads. Bad for users who follow 5000 people.

2. Push model (Fan-out on write)

When user A tweets, write the tweet into the inbox of every follower. Reads are instant. Bad for celebrities — Justin Bieber would need 300M inserts per tweet.

3. Hybrid (what Twitter actually does)

Push for normal users. Pull for celebrities. When you open feed, merge your pre-computed inbox with on-the-fly pulls from celebrities you follow.

Architecture

Tweet → API → Tweet Service → Cassandra (tweets) │ └→ Kafka → Fanout Service │ └→ Redis (per-user timeline) Feed read → Timeline Service → Redis → Merge with celebrity pull

Key components

  • Tweets stored in Cassandra, partitioned by user_id.
  • Each user's timeline kept as a Redis list of tweet IDs (last 800).
  • Media on S3 + CDN.
  • Search via Elasticsearch.

Design WhatsApp / Chat

Functional needs

  • 1:1 messaging, group chats
  • Online/last-seen status
  • Read receipts
  • Media (photo, video)
  • Push notifications when offline

Connection model

Each user maintains a persistent WebSocket connection to a "Chat Server". On message, server routes it via a message broker (Kafka) to the recipient's chat server, which pushes it down their WebSocket.

User A ── WS ──→ Chat Server 1 │ Kafka topic │ User B ←── WS ── Chat Server 2

If recipient offline

Message stored in a per-user inbox in Cassandra. When user reconnects → fetch all undelivered messages → deliver → mark delivered.

Online presence

Heartbeat every 10 seconds. No heartbeat for 30s → mark offline. Status changes pushed via pub/sub.

Read receipts

Client sends ACK when message displayed. Server updates message state (sent → delivered → read).

Media

Client uploads to blob storage (S3), gets URL, sends URL in the chat message. Receivers download from CDN.

End-to-end encryption

Signal Protocol — keys exchanged between clients; server only sees ciphertext. Servers never store decryption keys.

Design Uber / Ride-share

Core challenge

Match nearby drivers to riders in real time at city scale.

Location updates

Each driver sends GPS coords every 4 seconds via WebSocket. Stored in an in-memory geospatial DB (Redis GEO or QuadTree).

Geo-sharding

Divide the world into a grid (using Google's S2 cells or QuadTree). Each cell handled by a server. When a rider requests a ride, query nearby cells for drivers.

Architecture

Driver app ──→ Location Service ──→ Redis GEO ↑ Rider request → Dispatch ──────────────┘ │ └→ Match driver (closest k drivers, ETA, surge) └→ Notify driver via WebSocket └→ On accept → Trip Service │ └→ Kafka (events) └→ Payment, ETA, Analytics

Components

  • Location Service: ingests driver pings.
  • Dispatch Service: matching algorithm. Considers distance, driver rating, surge pricing.
  • Trip Service: manages trip state machine (requested → accepted → started → ended).
  • Payment Service: async after ride.
  • Notification Service: push notifications via FCM/APNS.

Design Netflix / Video Streaming

Three big challenges

  1. Storing petabytes of video.
  2. Delivering it smoothly to 200M users worldwide.
  3. Personalizing recommendations.

Upload & encoding pipeline

Raw video → S3 → encoded into multiple resolutions (240p, 480p, 720p, 1080p, 4K) and multiple codecs (H.264, H.265, VP9). Each chunked into small ~10-second segments. Result: thousands of files per original video.

Streaming protocol

Adaptive Bitrate Streaming (HLS / DASH). Player downloads a manifest, then fetches 10-sec chunks. If network slows, it switches to a lower-quality chunk seamlessly.

CDN

Netflix runs its own CDN — Open Connect. Servers physically placed inside ISP datacenters. 95% of traffic served from within your ISP's network.

Recommendations

Offline ML pipeline (Spark / TensorFlow) processes user watch history nightly. Output: ranked list of titles per user, stored in Cassandra. The homepage fetches these pre-computed ranks instantly.

Watch state

Position in the video saved every 30 seconds to a key-value store, so you can resume on any device.

LLD: Elevator System

Requirements

  • Building with N floors, M elevators.
  • Up/Down buttons on each floor; floor buttons inside elevator.
  • Efficient dispatcher — minimize wait time.

Classes

enum Direction { UP, DOWN, IDLE }
enum ElevatorState { MOVING, STOPPED, OUT_OF_SERVICE }

class Elevator {
    int id;
    int currentFloor;
    Direction direction;
    ElevatorState state;
    TreeSet<Integer> upRequests   = new TreeSet<>();
    TreeSet<Integer> downRequests = new TreeSet<>(Collections.reverseOrder());

    void addRequest(int floor);
    void step();   // advance one floor
}

class ElevatorController {
    List<Elevator> elevators;
    void requestElevator(int floor, Direction dir);
    Elevator pickBestElevator(int floor, Direction dir);
}

class Floor {
    int number;
    Button up, down;
}

Algorithm — LOOK (the real one used)

Elevator goes up servicing all up-requests above it, then reverses and services all down-requests below it. Much better than naïve FIFO.

Patterns used

  • State — elevator's behavior changes by state.
  • Command — each button press is a request command.
  • Observer — elevator notifies display panels when floor changes.

Final Interview Checklist

Before the interview

Memorize the back-of-envelope numbers in Chapter 04. Practice drawing the headline diagram (CDN → LB → App → Cache → DB → Queue) until you can do it in 30 seconds.

During the interview — the 10 commandments

  1. Don't jump into solutions. Ask clarifying questions first.
  2. Always confirm scale before designing (DAU, QPS, storage).
  3. Draw as you talk — don't go silent.
  4. Start simple, then scale. Single server first, then introduce caches, queues, replicas.
  5. Name your components precisely (Redis, Kafka, Cassandra — not "a database").
  6. Mention trade-offs. "I'm choosing Cassandra over MongoDB because we need write-heavy and partition tolerance."
  7. Plan for failure. What if the cache dies? What if a region goes down?
  8. Think about consistency. Strong? Eventual? Why?
  9. Discuss monitoring — metrics, logs, alerts.
  10. Manage time. Don't spend 30 minutes on schema and skip the architecture.

Red flags to avoid

  • Saying "I'd use a database" without naming which kind and why.
  • Designing for 10 users when the question says 100M.
  • Ignoring read:write ratio.
  • Forgetting CAP — every interviewer asks about consistency.
  • Adding microservices when not needed.
  • No mention of caching layer.

Phrases that impress

  • "Let me first clarify the requirements before estimating."
  • "Given the read:write ratio of X:1, I'll optimize the read path with..."
  • "There's a trade-off here — we can go AP for availability or CP for consistency. Given the use case, I'd choose..."
  • "This is a single point of failure — let me add redundancy."
  • "I'd shard by user_id using consistent hashing to allow elastic scaling."
Final word

System design isn't memorization. It's a conversation about trade-offs. Master these 40 chapters and you'll have the vocabulary to hold that conversation with any interviewer at any MNC. Now go ship it.
