Every concept in the Java language explained in plain English — with code, gotchas, and the interview questions you'll actually be asked. From your first System.out.println to virtual threads.

Java Architecture

Java was designed in 1995 by James Gosling at Sun Microsystems with one big idea: "Write Once, Run Anywhere". Your code should run on any machine without modification. That's why Java is everywhere — Android, banking, web services, big data, enterprise systems.

JDK vs JRE vs JVM

TermStands forWhat it does
JDKJava Development KitFor developers — includes compiler (javac), JRE, tools.
JREJava Runtime EnvironmentFor running Java apps — includes JVM + standard libraries.
JVMJava Virtual MachineThe actual runtime that executes .class bytecode.

Rule of thumb: JDK contains JRE, JRE contains JVM. To write Java → install JDK. To just run Java → JRE is enough.

How Java code runs

Hello.java ──── javac ────► Hello.class ──── JVM ────► Output (source) (bytecode) (interprets + JIT compiles)

The same .class file runs on Windows, Mac, Linux — each OS has its own JVM that translates bytecode to native machine code. That's the magic.

Your first program

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}

Save as Hello.java, then compile and run:

$ javac Hello.java     # produces Hello.class
$ java Hello           # runs it
Hello, World!
Why the verbose syntax?

public class Hello — every program needs at least one class. main is the entry point. String[] args holds command-line arguments. static means JVM can call it without creating an object first. Yes, it's verbose. That's Java.

Common Interview Q
Is Java compiled or interpreted?
Both. Source → bytecode by javac (compilation), then JVM interprets bytecode and the JIT (Just-In-Time compiler) further compiles hot bytecode to native code at runtime.
Why is Java platform-independent?
Because bytecode targets the JVM, not the OS. Different OSes have different JVMs that translate the same bytecode.
What is JIT?
Just-In-Time compiler — part of the JVM. Identifies frequently-run "hot" code and compiles it to native instructions for faster execution.

Variables & Data Types

Java is statically typed — every variable's type is fixed at declaration. There are two categories:

1. Primitive types (8 total)

2. Reference types

Anything else — String, arrays, objects, custom classes. These store a reference (pointer) to data on the heap, not the data itself.

// Primitives — stored on the stack
int age = 25;
double salary = 75000.50;
boolean isActive = true;
char grade = 'A';

// References — stored on the heap; variable holds the address
String name = "Alice";
int[] numbers = {1, 2, 3};

Wrapper classes

Every primitive has an object equivalent: Integer, Double, Character, Boolean... Collections (like List) can't hold primitives, so wrappers fill the gap.

int prim = 42;
Integer wrapped = 42;        // autoboxing  (int → Integer)
int unwrapped = wrapped;     // unboxing    (Integer → int)

Type casting

// Implicit (widening) — safe, automatic
int i = 100;
long l = i;                  // int fits in long

// Explicit (narrowing) — may lose data
double d = 9.78;
int x = (int) d;             // x = 9, decimal dropped

var keyword (Java 10+)

var list = new ArrayList<String>();    // compiler infers type
var name = "Alice";                     // String
var count = 5;                          // int
Watch out

var is not "dynamic" — it's just less typing. The type is still fixed at compile time. var x = 5; creates an int; you can't later say x = "hello".

Common Interview Q
Difference between int and Integer?
int is a primitive (4 bytes, value on stack). Integer is an object wrapper (reference on heap, has methods like compareTo). Integer can be null; int cannot.
What is autoboxing?
Automatic conversion of primitive to its wrapper (or vice versa) by the compiler. Integer x = 5; is silently Integer x = Integer.valueOf(5);
Why does Integer a = 127; Integer b = 127; a == b return true, but a = 128; b = 128; a == b returns false?
Java caches Integer values from −128 to 127. Same values in that range share the same object → == matches. Outside that range, new objects are created → == compares references. Always use .equals() for value comparison.

Short-circuit evaluation

&& and || stop evaluating as soon as the result is determined.

if (obj != null && obj.method()) { ... }
// If obj is null, obj.method() is never called → no NullPointerException

++ pre vs post

int x = 5;
int a = x++;     // a = 5, then x becomes 6 (post-increment)
int b = ++x;     // x becomes 7, then b = 7 (pre-increment)
Integer division truncates

5 / 2 == 2 in Java (not 2.5). Use 5.0 / 2 or cast: (double) 5 / 2 → 2.5.

Common Interview Q
What's the difference between & and &&?
& is bitwise AND on integers and non-short-circuit boolean AND. && is short-circuit logical AND — stops evaluating once result is known.
What is >>>?
Unsigned right shift. Fills high bits with 0 regardless of sign. -1 >> 1 stays negative; -1 >>> 1 becomes a large positive number.

Control Flow

if / else

if (age < 13)        System.out.println("Child");
else if (age < 20)   System.out.println("Teenager");
else                 System.out.println("Adult");

switch (classic)

switch (day) {
    case "MON":
    case "TUE":
        System.out.println("Start of week");
        break;
    case "FRI":
        System.out.println("Almost weekend");
        break;
    default:
        System.out.println("Other day");
}

switch expression (Java 14+) — cleaner

String result = switch (day) {
    case "MON", "TUE" -> "Start of week";
    case "FRI"        -> "Almost weekend";
    default           -> "Other day";
};

Loops

// classic for
for (int i = 0; i < 10; i++) { ... }

// while
while (condition) { ... }

// do-while — runs at least once
do { ... } while (condition);

// enhanced for-each
for (String name : names) {
    System.out.println(name);
}

break, continue, labeled loops

outer:
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        if (matrix[i][j] == target) {
            break outer;          // exits both loops
        }
    }
}
Common Interview Q
What types can switch work with?
byte, short, int, char, their wrappers, String (Java 7+), enums, and (Java 21+) patterns/sealed types.
Difference between while and do-while?
do-while always executes the body at least once because the condition is checked at the end.

Methods

A method is a named, reusable block of code. Anatomy:

public static int add(int a, int b) {     // signature
    return a + b;                          // body
}
//     ^      ^   ^     ^------- parameters
//     |      |   +------------ return type
//     |      +---------------- method name
//     +----------------------- access modifier(s)

Method overloading

Same method name, different parameter list. Resolved at compile time.

int add(int a, int b)               { return a + b; }
double add(double a, double b)      { return a + b; }
int add(int a, int b, int c)        { return a + b + c; }

Varargs (variable arguments)

int sum(int... nums) {              // nums is an int[] inside
    int total = 0;
    for (int n : nums) total += n;
    return total;
}
sum(1, 2);          // 3
sum(1, 2, 3, 4);    // 10
sum();              // 0

Pass by value (always!)

Java is always pass by value. For primitives, the value is copied. For objects, the reference is copied — so both variables point to the same object.

void changePrimitive(int x) { x = 100; }
void changeObject(int[] arr) { arr[0] = 100; }

int n = 5;
changePrimitive(n);
System.out.println(n);          // still 5

int[] arr = {1, 2, 3};
changeObject(arr);
System.out.println(arr[0]);     // 100 — same array on heap
Common misconception

Java is NOT "pass by reference". It's pass by value of references for objects. If you reassign the parameter (arr = new int[]{9, 9}), the caller's array doesn't change.

Common Interview Q
Is Java pass-by-value or pass-by-reference?
Always pass-by-value. For primitives, the value is copied. For objects, the reference (memory address) is copied — both refer to the same heap object, but they're independent variables.
Can main be overloaded?
Yes — but only the standard public static void main(String[] args) is JVM's entry point. Other overloads are normal methods.

Arrays

A fixed-size collection of same-type elements stored contiguously. Once created, you can't change the size.

// Declare and initialize
int[] nums = new int[5];                 // {0, 0, 0, 0, 0}
int[] vals = {10, 20, 30};
String[] words = new String[]{"a", "b"};

// Access & modify
nums[0] = 42;
int x = vals[1];

// Length (NOT a method — it's a field)
System.out.println(vals.length);         // 3

// Iterate
for (int n : nums) System.out.println(n);

Multi-dimensional arrays

int[][] grid = new int[3][4];            // 3 rows, 4 columns
int[][] matrix = {{1, 2}, {3, 4}, {5, 6}};
System.out.println(matrix[1][0]);        // 3

The Arrays utility class

int[] a = {3, 1, 4, 1, 5, 9};
Arrays.sort(a);                          // {1, 1, 3, 4, 5, 9}
Arrays.fill(a, 0);                       // {0, 0, 0, 0, 0, 0}
int[] b = Arrays.copyOf(a, 10);          // expand to 10 elements
boolean eq = Arrays.equals(a, b);
String s = Arrays.toString(a);           // "[0, 0, 0, 0, 0, 0]"
List<Integer> list = Arrays.asList(1, 2, 3);  // fixed-size view
Pitfall — Arrays.asList

Arrays.asList(arr) returns a fixed-size list backed by the array — you can't add() or remove(). Wrap in new ArrayList<>(Arrays.asList(arr)) for a true mutable list.

Common Interview Q
Difference between array and ArrayList?
Array: fixed size, stores primitives or objects, contiguous memory. ArrayList: dynamic size, only objects, backed internally by an array that grows.
Default values of array elements?
Numeric: 0; boolean: false; char: '\u0000'; references: null.

Strings

The String class is the most-used class in Java. Two critical facts:

  1. Strings are immutable — once created, can't be changed. Operations like concat() or + create new String objects.
  2. String literals live in the String Pool — a special area of the heap. Identical literals share the same object.

String pool magic

String a = "hello";          // goes into the pool
String b = "hello";          // reuses the pool entry
String c = new String("hello");  // forced new object on heap

System.out.println(a == b);          // true  (same pool reference)
System.out.println(a == c);          // false (different objects)
System.out.println(a.equals(c));     // true  (same content)
Always use .equals() for strings

== compares references (memory addresses). .equals() compares content. This is the #1 Java bug for beginners.

Common String methods

String s = "Hello, World";
s.length();              // 12
s.charAt(0);             // 'H'
s.substring(7);          // "World"
s.substring(7, 12);      // "World"
s.indexOf("World");      // 7
s.toLowerCase();         // "hello, world"
s.toUpperCase();
s.trim();                // strip leading/trailing whitespace
s.replace("World", "Java");
s.split(", ");           // {"Hello", "World"}
s.contains("World");     // true
s.startsWith("Hello");
String.join("-", "a", "b", "c");   // "a-b-c"

StringBuilder — for mutation

Because String is immutable, repeated concatenation in a loop creates many garbage objects. Use StringBuilder instead.

// BAD — creates n new strings
String s = "";
for (int i = 0; i < n; i++) s += i;       // O(n²)

// GOOD — single mutable buffer
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) sb.append(i); // O(n)
String result = sb.toString();
StringStringBuilderStringBuffer
MutableNoYesYes
Thread-safeYes (immutable)NoYes
Speed—FastSlower (locks)
Common Interview Q
Why is String immutable?
Three reasons: (1) Thread-safety — no synchronization needed. (2) String pool — sharing is only safe if values can't change. (3) HashMap key safety — if a key's hashCode could change, the map would break.
String s = "hi"; vs new String("hi")?
The literal goes to the String pool (reused). new String("hi") always creates a fresh heap object even if "hi" exists in the pool.
StringBuilder vs StringBuffer?
Same API. StringBuffer is synchronized (thread-safe but slower). StringBuilder is faster, single-threaded. Default to StringBuilder.

Classes & Objects

A class is a blueprint. An object is an instance — a real thing built from that blueprint. A class defines what an object knows (fields) and what it can do (methods).

public class Car {
    // fields (state)
    String color;
    int speed;

    // method (behavior)
    void accelerate() {
        speed += 10;
    }
}

// create an object
Car myCar = new Car();
myCar.color = "Red";
myCar.accelerate();
System.out.println(myCar.speed);   // 10

Memory picture

Stack Heap ──────── ───────────────── myCar ─────────────► Car { color: "Red" speed: 10 }

The local variable myCar on the stack holds a reference (address). The actual object lives on the heap.

Common Interview Q
What are the 4 pillars of OOP?
Encapsulation, Inheritance, Polymorphism, Abstraction. (Some add: composition.)
Class vs Object?
Class = blueprint (template). Object = instance built from that blueprint. One class, many objects.

Constructors, this, static

Constructors

A special method called when an object is created. Same name as the class, no return type. If you don't write one, Java provides a default no-arg constructor.

public class Car {
    String color;
    int speed;

    // no-arg constructor
    Car() {
        this.color = "White";
        this.speed = 0;
    }

    // parameterized
    Car(String color, int speed) {
        this.color = color;
        this.speed = speed;
    }

    // constructor calling another constructor
    Car(String color) {
        this(color, 0);          // delegate to other constructor
    }
}

The this keyword

Reference to the current object. Used to:

  • Disambiguate field vs parameter with the same name: this.color = color;
  • Pass current object to other methods.
  • Call another constructor: this(args);

The static keyword

Belongs to the class, not an instance. One copy shared by all objects.

public class Counter {
    static int totalCount = 0;        // class variable
    int instanceCount = 0;            // instance variable

    static void reset() {             // class method
        totalCount = 0;
    }
}

Counter.totalCount;                   // accessed via class name
Counter.reset();                      // no object needed

Static blocks

public class Config {
    static Map<String, String> settings;

    static {                          // runs once when class loads
        settings = new HashMap<>();
        settings.put("env", "prod");
    }
}
Common Interview Q
Can a constructor be private?
Yes. Used in Singleton pattern and utility classes that shouldn't be instantiated.
Can static methods access non-static fields?
No. Static methods belong to the class; instance fields belong to specific objects, so the question "which instance's field?" has no answer.
Can constructors be inherited?
No. But a subclass constructor must call a parent constructor (via super(...) or implicitly super()).

Inheritance

A class can extend another class — inheriting its fields and methods. The child is a specialized version of the parent.

class Animal {
    String name;
    void eat()  { System.out.println(name + " is eating"); }
    void sleep(){ System.out.println(name + " is sleeping"); }
}

class Dog extends Animal {
    void bark() { System.out.println(name + " says Woof!"); }
}

Dog d = new Dog();
d.name = "Rex";
d.eat();      // inherited from Animal
d.bark();     // defined in Dog

The super keyword

class Dog extends Animal {
    Dog(String name) {
        super();                  // calls Animal's constructor (implicit)
        this.name = name;
    }

    @Override
    void eat() {
        super.eat();              // call parent's version
        System.out.println("...specifically dog food");
    }
}

The Object class

Every Java class implicitly extends Object. That's why every object has toString(), equals(), hashCode(), getClass().

Single inheritance only

Java allows one parent class. To get multiple inheritance, use interfaces.

final classes / methods

final class String { ... }            // cannot be extended
class Parent { final void doIt(){} }  // cannot be overridden
Common Interview Q
Why doesn't Java support multiple inheritance of classes?
To avoid the "diamond problem" — ambiguity when two parents have the same method. Java sidesteps this by allowing multiple interface inheritance instead.
What is super()?
A call to the parent class's constructor. Java inserts it implicitly at the top of every subclass constructor if you don't write it.
Difference between IS-A and HAS-A?
IS-A is inheritance: Dog IS-A Animal. HAS-A is composition: Car HAS-A Engine. Prefer composition when behavior changes; use inheritance only for true "is-a" relationships.

Polymorphism

"Polymorphism" = many shapes. The same method call behaves differently depending on the actual object. Two flavors:

1. Compile-time (overloading)

Same method name, different parameters. Resolved by the compiler.

void print(int x)    { System.out.println("int: " + x); }
void print(double x) { System.out.println("double: " + x); }
void print(String x) { System.out.println("string: " + x); }

2. Runtime (overriding) — the real polymorphism

Subclass redefines a parent method. Resolved at runtime based on the actual object type.

class Animal {
    void sound() { System.out.println("Some sound"); }
}
class Dog extends Animal {
    @Override
    void sound() { System.out.println("Woof"); }
}
class Cat extends Animal {
    @Override
    void sound() { System.out.println("Meow"); }
}

Animal a;
a = new Dog();  a.sound();   // "Woof"   — actual object decides
a = new Cat();  a.sound();   // "Meow"
Why this is powerful

You can write one method that processes List<Animal> without caring which species. Each one will produce its own sound automatically. That's polymorphism's superpower — code that works with abstractions, not specifics.

Rules for overriding

  • Same method signature (name + parameters).
  • Same or covariant return type.
  • Access modifier can't be more restrictive than parent.
  • Can't override final, static, or private methods.
  • Use @Override annotation — catches typos at compile time.

Common Interview Q
Can static methods be overridden?
No. Static methods are bound to the class at compile time. Redeclaring a static method in a subclass is "hiding", not overriding.
Can constructors be overloaded? Overridden?
Overloaded yes; overridden no — constructors aren't inherited.

Abstract Classes & Interfaces

Both let you define what a class must do without saying how. The differences matter at interview time.

Abstract class

abstract class Shape {
    String color;                              // can have state

    Shape(String color) { this.color = color; } // can have constructor

    void describe() {                          // concrete method
        System.out.println("A " + color + " shape");
    }

    abstract double area();                    // subclasses MUST implement
}

class Circle extends Shape {
    double r;
    Circle(String c, double r) { super(c); this.r = r; }

    @Override
    double area() { return Math.PI * r * r; }
}

Interface

interface Drawable {
    void draw();                               // implicit public abstract

    default void log() {                       // Java 8 — default method
        System.out.println("Drawing...");
    }

    static Drawable empty() {                  // Java 8 — static method
        return () -> {};
    }
}

class Square implements Drawable {
    @Override
    public void draw() { System.out.println("□"); }
}

Multiple interfaces

class Duck extends Bird implements Swimmer, Flyer {
    // must implement abstract methods from both interfaces
}

The big comparison

Abstract ClassInterface
State (fields)Yes — any fieldsOnly public static final (constants)
ConstructorsYesNo
Method typesAbstract + concreteAbstract, default, static, private (9+)
Multiple inheritanceNo (one parent)Yes (many interfaces)
Use whenStrong is-a + shared stateCapability / contract
Rule of thumb

Use interface when defining a capability (Comparable, Runnable, Serializable). Use abstract class when sharing partial implementation among closely related classes.

Common Interview Q
Why were default methods added in Java 8?
To extend existing interfaces (like Collection) with new methods (like stream()) without breaking every implementation.
Can an interface extend another interface?
Yes — even multiple interfaces: interface A extends B, C { }.
Can interfaces have private methods?
Since Java 9, yes — to share code between default methods.

Encapsulation & Access Modifiers

Encapsulation = hiding internal state and exposing it only through controlled methods. It's the "don't touch my privates" principle.

public class BankAccount {
    private double balance;          // hidden

    public double getBalance() {     // controlled read
        return balance;
    }

    public void deposit(double amount) {  // controlled write
        if (amount <= 0) throw new IllegalArgumentException();
        balance += amount;
    }
}

You can never set balance to a negative number from outside. The class controls its own invariants.

The 4 access modifiers

Default (no modifier) is also called "package-private".

Common Interview Q
Why use getters/setters instead of public fields?
Control: validate inputs, log access, change internal representation later without breaking callers. Java's frameworks (JPA, Spring, JSON libs) also rely on the JavaBean conventions.
Can a top-level class be private?
No. Only public or default. Nested classes can be private.

Packages

A package is a namespace — a folder for grouping related classes. Avoids name collisions across libraries.

package com.acme.banking;          // first non-comment line of file

import java.util.List;
import java.util.ArrayList;
import java.util.*;                // wildcard — all classes in java.util
import static java.lang.Math.PI;   // static import: use PI directly

public class Account { ... }

Standard packages worth knowing

  • java.lang — String, Math, Thread, Object. Auto-imported.
  • java.util — Collections, Date, Scanner, Random.
  • java.io — File, BufferedReader, InputStream.
  • java.nio — Modern non-blocking I/O.
  • java.time — Modern date/time (Java 8+).
  • java.util.concurrent — Threading primitives.
  • java.util.stream — Stream API.
Common Interview Q
Why are packages important?
They prevent class-name collisions, control access (package-private), and organize large codebases logically.

Inner Classes & Enums

Four flavors of nested classes

class Outer {
    // 1. Static nested class — no link to outer instance
    static class StaticNested { ... }

    // 2. Inner class — implicit reference to Outer instance
    class Inner { ... }

    void method() {
        // 3. Local inner class — defined inside a method
        class Local { ... }

        // 4. Anonymous inner class — no name, used once
        Runnable r = new Runnable() {
            @Override
            public void run() { System.out.println("Hi"); }
        };
    }
}

Most anonymous inner classes today are replaced by lambdas:

Runnable r = () -> System.out.println("Hi");

Enums

A fixed set of named constants. Type-safe and feature-rich in Java.

enum Day {
    MON, TUE, WED, THU, FRI, SAT, SUN;
}

Day today = Day.MON;
if (today == Day.SAT || today == Day.SUN) { ... }

Enums with state and methods

enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    EARTH(5.976e+24, 6.37814e6);

    private final double mass;
    private final double radius;

    Planet(double mass, double radius) {
        this.mass = mass;
        this.radius = radius;
    }

    double surfaceGravity() {
        return 6.67300E-11 * mass / (radius * radius);
    }
}

double g = Planet.EARTH.surfaceGravity();
Common Interview Q
Can enums implement interfaces?
Yes. Each constant can even override methods if needed.
Are enums singletons?
Each constant is a single instance — JVM guarantees it. That's why enum is the recommended way to implement the Singleton pattern.

Exception Handling

Things go wrong — files don't exist, networks fail, arrays go out of bounds. Java uses exceptions to signal these problems instead of crashing.

The hierarchy

Throwable / \ Error Exception (JVM / \ problems) Checked RuntimeException (must (unchecked — handle) NullPointer, ArrayIndexOutOfBounds, ArithmeticException, ...)

Checked vs Unchecked

CheckedUnchecked (Runtime)
ExampleIOException, SQLExceptionNullPointerException, IllegalArgumentException
When to useRecoverable, expected failuresProgramming errors / bugs
Must handle?Yes (compile error if not)No (but you can)

try / catch / finally

try {
    String content = Files.readString(Path.of("data.txt"));
} catch (IOException e) {
    System.err.println("Couldn't read file: " + e.getMessage());
} catch (Exception e) {              // general fallback
    e.printStackTrace();
} finally {
    System.out.println("This always runs");
}

Multi-catch (Java 7+)

try {
    ...
} catch (IOException | SQLException e) {
    // handle both
}

try-with-resources (auto-close)

try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    return br.readLine();
}
// br.close() called automatically, even on exception

throw vs throws

// throw — actually throw an exception
if (age < 0) throw new IllegalArgumentException("age can't be negative");

// throws — declare that a method may throw a checked exception
public String readFile(String path) throws IOException { ... }

Custom exception

public class InsufficientFundsException extends RuntimeException {
    public InsufficientFundsException(String msg) {
        super(msg);
    }
}
Common Interview Q
Difference between Error and Exception?
Error = JVM-level problems (OutOfMemoryError, StackOverflowError) — usually unrecoverable. Exception = application-level — should be handled.
Will finally run if try has a return?
Yes. finally runs even if try or catch return — almost. It does not run if JVM exits via System.exit().
When use checked vs unchecked?
Checked for recoverable conditions outside the program's control (I/O, network). Unchecked for programming mistakes (null dereferencing, bad arguments) — modern style favors unchecked.

An ordered collection that allows duplicates and indexed access.

Common Interview Q
When does ArrayList grow?
When size == capacity. New capacity = oldCapacity + (oldCapacity >> 1), i.e. 1.5×. Elements are copied to the new array.
Is ArrayList synchronized?
No. For thread safety, wrap with Collections.synchronizedList() or use CopyOnWriteArrayList.

Set

A collection of unique elements. No duplicates allowed.

HashSet — fastest, no order

Set<String> set = new HashSet<>();
set.add("Alice");
set.add("Bob");
set.add("Alice");          // ignored, already present
set.contains("Alice");     // true, O(1)
set.size();                // 2

Internally backed by a HashMap. O(1) add, contains, remove.

LinkedHashSet — insertion order

Same as HashSet but maintains the order in which you added elements.

TreeSet — sorted

Backed by a Red-Black Tree. O(log n) operations. Elements must be Comparable or you must pass a Comparator.

TreeSet<Integer> ts = new TreeSet<>();
ts.add(3); ts.add(1); ts.add(2);
System.out.println(ts);    // [1, 2, 3]
ts.first();                // 1
ts.last();                 // 3
ts.floor(2);               // 2  (largest ≤ 2)
ts.ceiling(2);             // 2  (smallest ≥ 2)
Common Interview Q
How does HashSet ensure uniqueness?
By using hashCode() and equals(). Adding an element first hashes it to a bucket; if any existing element in that bucket equals the new one, it's rejected.
Can a HashSet contain null?
Yes — exactly one null. TreeSet can't (it tries to compare null and throws NPE).

Map (HashMap deep dive)

A collection of key → value pairs. Keys are unique; values can repeat.

HashMap basics

Map<String, Integer> ages = new HashMap<>();
ages.put("Alice", 30);
ages.put("Bob", 25);
ages.get("Alice");                       // 30
ages.containsKey("Alice");               // true
ages.remove("Bob");
ages.size();
ages.getOrDefault("Charlie", 0);         // 0
ages.putIfAbsent("Alice", 99);           // no-op, key exists
ages.computeIfAbsent("Dan", k -> 40);    // insert if missing
ages.merge("Alice", 1, Integer::sum);    // ages["Alice"] += 1

for (var e : ages.entrySet()) {
    System.out.println(e.getKey() + " = " + e.getValue());
}

How HashMap works internally

  1. Default capacity = 16 buckets, load factor = 0.75.
  2. put(key, value) computes hash(key), picks a bucket.
  3. If the bucket is empty → new entry inserted.
  4. If occupied (collision) → linked list of entries in that bucket. Equal keys are replaced.
  5. When a bucket has 8+ entries → list is converted to a Red-Black tree (Java 8+) — improves worst-case from O(n) to O(log n).
  6. When size > capacity × loadFactor → resize: double the bucket array, re-hash all entries.

Why hashCode/equals matter for keys

If you use a custom object as a key but don't override hashCode() and equals(), two "equal" objects will land in different buckets — your lookups will fail silently.

Common Interview Q
Default initial capacity of HashMap?
16 buckets, load factor 0.75 → first resize at size 12.
Why does HashMap allow null keys?
It uses hash 0 for null and stores it in bucket 0. Only one null key allowed.
HashMap vs ConcurrentHashMap?
HashMap is unsynchronized. ConcurrentHashMap allows safe concurrent access with high throughput by locking only segments (or individual buckets in newer versions), not the whole map.

Queue, Deque & PriorityQueue

Queue interface — FIFO

Queue<Integer> q = new ArrayDeque<>();
q.offer(1);                  // enqueue (returns false if full)
q.offer(2);
q.poll();                    // dequeue (returns null if empty)
q.peek();                    // look without removing

Deque — double-ended queue

Deque<Integer> d = new ArrayDeque<>();
d.addFirst(1); d.addLast(2);
d.peekFirst(); d.peekLast();
d.pollFirst(); d.pollLast();

// Also great as a stack
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1); stack.push(2);
stack.pop();                 // 2
stack.peek();                // 1

PriorityQueue — min-heap by default

PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.offer(5); pq.offer(1); pq.offer(3);
pq.poll();                   // 1 (smallest)

// Max-heap
PriorityQueue<Integer> max = new PriorityQueue<>(Comparator.reverseOrder());

// Custom: sort tasks by priority
PriorityQueue<Task> tasks = new PriorityQueue<>(
    (a, b) -> a.priority - b.priority
);
Java tip

Always prefer ArrayDeque over LinkedList for queue/stack usage. It's faster (cache-friendly) and uses less memory.

Common Interview Q
Is Java's PriorityQueue thread-safe?
No. Use PriorityBlockingQueue for concurrent priority access.
Why is Stack class discouraged?
It extends Vector (synchronized, slow) and exposes ill-suited APIs. Deque/ArrayDeque is the modern replacement.

Iterators & Comparators

Iterator

List<String> list = new ArrayList<>(List.of("a", "b", "c"));
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.equals("b")) it.remove();      // safe removal during iteration
}
ConcurrentModificationException

Modifying a collection (via list.remove()) while iterating throws this exception. Use the iterator's own .remove() method, or collect items to remove and delete after the loop.

Comparable — natural order

Implement when an object has an "obvious" ordering. Used by Arrays.sort(), TreeSet, etc.

class Employee implements Comparable<Employee> {
    String name;
    int salary;

    @Override
    public int compareTo(Employee other) {
        return this.salary - other.salary;   // ascending by salary
    }
}

Comparator — multiple orderings

// Old style
Comparator<Employee> byName = new Comparator<>() {
    @Override
    public int compare(Employee a, Employee b) {
        return a.name.compareTo(b.name);
    }
};

// Modern with lambda
Comparator<Employee> byName2 = (a, b) -> a.name.compareTo(b.name);

// Even better with method references
Comparator<Employee> bySalary = Comparator.comparingInt(e -> e.salary);
Comparator<Employee> bySalaryDesc = bySalary.reversed();
Comparator<Employee> chain = Comparator.comparing((Employee e) -> e.name)
                                       .thenComparingInt(e -> e.salary);
Common Interview Q
Comparable vs Comparator?
Comparable defines natural ordering inside the class. Comparator is external — lets you define different orderings without touching the class.
Why does a.salary - b.salary sometimes fail?
Integer overflow. If salaries can be very large positive and very large negative, subtraction overflows. Use Integer.compare(a, b) or Comparator.comparingInt.

Generics

Generics let you write code that works with any type while keeping type safety. Before Java 5, every collection held Object and you had to cast everywhere.

// Before generics — runtime casting, error-prone
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);

// With generics — compile-time safety
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);          // no cast

Generic class

public class Box<T> {
    private T content;
    public void set(T content) { this.content = content; }
    public T get() { return content; }
}

Box<String> sb = new Box<>();
sb.set("hi");

Generic method

public static <T> T firstNonNull(T a, T b) {
    return a != null ? a : b;
}

Bounded types

// Upper bound — T must be a Number (or subtype)
public <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue();
    return total;
}

Wildcards: ?, ? extends, ? super

List<?> unknown;                    // unknown type — read-only
List<? extends Number> numbers;     // some subtype of Number — for reading
List<? super Integer> ints;         // some supertype of Integer — for writing

PECS — Producer Extends, Consumer Super

  • If you're reading values out of a collection → ? extends T.
  • If you're writing values into a collection → ? super T.

Type erasure

Generics exist only at compile time. At runtime, List<String> and List<Integer> are both just List. That's why you can't do new T[10] or instanceof List<String>.

Common Interview Q
What's type erasure?
Compiler removes generic type info after type-checking, replacing T with Object (or its upper bound). This preserves backwards compatibility but limits what you can do with generics at runtime.
Difference between List<Object> and List<?>?
List<Object> means "a list of Object" — you can add anything. List<?> means "a list of unknown type" — you can only add null and read as Object.

Lambda Expressions

Java 8 brought lambdas — anonymous functions you can pass around like values. A massive boost in expressiveness.

// Old style — anonymous inner class
Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("Hi");
    }
};

// Lambda — same thing, much shorter
Runnable r = () -> System.out.println("Hi");

Syntax forms

() -> "Hello"                          // no args
x  -> x * 2                            // one arg (no parens)
(int x) -> x * 2                       // explicit type
(x, y) -> x + y                        // multiple args
(x, y) -> {                            // multi-statement body
    int z = x + y;
    return z * 2;
}

Effectively final capture

Lambdas can use local variables from the enclosing scope, but only if those variables aren't modified after capture (i.e., effectively final).

String prefix = "Mr. ";
names.forEach(name -> System.out.println(prefix + name));
// prefix = "Ms. ";  // ← would cause compile error
Common Interview Q
Lambda vs anonymous inner class?
Lambda is concise, has no separate this, no shadowing. Anonymous inner class can have multiple methods and its own state. Lambdas only work for functional interfaces (single abstract method).
What does this refer to inside a lambda?
The enclosing class — not the lambda itself. (In anonymous inner classes, this referred to the inner instance.)

A functional interface has exactly one abstract method. The lambda implements that method.

@FunctionalInterface

Validator<String> isEmail = s -> s.contains("@");
isEmail.validate("a@b.com");          // true

Stream API

Streams let you process collections functionally — declaratively rather than with explicit loops. They're lazy, composable, and parallel-ready.

// Imperative
List<String> result = new ArrayList<>();
for (String name : names) {
    if (name.startsWith("A")) {
        result.add(name.toUpperCase());
    }
}

// Functional
List<String> result = names.stream()
    .filter(n -> n.startsWith("A"))
    .map(String::toUpperCase)
    .collect(Collectors.toList());

The three stages

  1. Source — collection, array, file, generator.
  2. Intermediate operations (return another Stream): filter, map, sorted, distinct, limit, skip, peek, flatMap.
  3. Terminal operation (consumes the stream): collect, forEach, count, reduce, findFirst, anyMatch, toArray.

Common patterns

// Filter + Map + Collect
List<Integer> evens = nums.stream()
    .filter(n -> n % 2 == 0)
    .map(n -> n * n)
    .collect(Collectors.toList());

// Sum
int total = nums.stream().mapToInt(Integer::intValue).sum();

// Group by
Map<String, List<Person>> byCity = people.stream()
    .collect(Collectors.groupingBy(Person::getCity));

// Count occurrences
Map<String, Long> freq = words.stream()
    .collect(Collectors.groupingBy(w -> w, Collectors.counting()));

// Join strings
String csv = names.stream().collect(Collectors.joining(", "));

// Find any
Optional<String> first = names.stream()
    .filter(n -> n.startsWith("A"))
    .findFirst();

// Sort
List<Person> sorted = people.stream()
    .sorted(Comparator.comparingInt(Person::getAge))
    .collect(Collectors.toList());

// Parallel
int total = nums.parallelStream().mapToInt(Integer::intValue).sum();
Streams are one-shot

Once a terminal operation is called, the stream is consumed. Calling another operation throws IllegalStateException. Create a new stream each time.

Common Interview Q
Are streams faster than loops?
Usually no — there's overhead. Streams are about expressiveness, not raw speed. Parallel streams can speed up CPU-bound work on large data, but careful — overhead and ordering issues hurt small datasets.
map vs flatMap?
map transforms each element to one value. flatMap transforms each element to a stream, then flattens all the streams into one (one-to-many).
Why is Stream.collect(toList()) preferred over toList() (the method)?
Both work since Java 16+. Stream.toList() returns an unmodifiable list; Collectors.toList() returns a modifiable one.

Optional & java.time

Optional — kill null

Wraps a value that might be absent. Forces you to handle the empty case explicitly.

Optional<User> user = findUser(id);

// Bad — defeats the purpose
if (user.isPresent()) return user.get();

// Good — declarative
return user.map(User::getEmail)
           .orElse("noemail@x.com");

// Other useful methods
user.ifPresent(System.out::println);
user.orElseGet(() -> defaultUser());
user.orElseThrow(() -> new NotFoundException());
Don't use Optional as field or parameter

Optional is meant for return types. Using it for fields adds runtime cost and complicates serialization.

Garbage Collection

You never free memory in Java — the GC reclaims unreachable objects. An object is unreachable when no live thread can access it through any chain of references.

The generational hypothesis

Most objects die young. So:

  • New objects → Young Generation (Eden space). Collected often, fast.
  • Survivors after several GCs → promoted to Old Generation. Collected rarely.

Major garbage collectors

  • Serial GC — single-threaded. Small apps.
  • Parallel GC — multi-threaded, throughput-focused. Default in Java 8.
  • G1 (Garbage First) — splits heap into regions, low-pause. Default since Java 9.
  • ZGC — ultra-low pause times (<10ms), Java 11+.
  • Shenandoah — similar to ZGC, Red Hat's implementation.
Common Interview Q
Can you force garbage collection?
You can suggest with System.gc() — the JVM may ignore it. There's no way to force.
What is a memory leak in Java?
Unintentionally keeping references to objects you no longer need (e.g., putting objects in static collections that grow forever, listener objects never removed, ThreadLocals not cleared). The GC can't collect them — leak.
finalize() — should you use it?
No. It's deprecated since Java 9 and was always unpredictable. Use try-with-resources and AutoCloseable instead.

equals(), hashCode(), toString()

Every class inherits these three from Object. The defaults are usually wrong for value-like objects.

The equals/hashCode contract

  1. Reflexive: x.equals(x) = true.
  2. Symmetric: x.equals(y) = y.equals(x).
  3. Transitive: if x.equals(y) and y.equals(z), then x.equals(z).
  4. Consistent: repeated calls give the same result.
  5. If a.equals(b), then a.hashCode() == b.hashCode(). (Most important!)

The reverse isn't required — two objects can have the same hashCode but not be equal (collision).

Implementation pattern

public class Point {
    private final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point p)) return false;    // Java 16+ pattern matching
        return x == p.x && y == p.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    @Override
    public String toString() {
        return "Point(" + x + ", " + y + ")";
    }
}
Why override both, not just equals

If you override equals but not hashCode, two "equal" objects hash to different buckets in a HashMap — and your map will have ghost duplicates that you can never retrieve.

Common Interview Q
What if two unequal objects have the same hashCode?
That's a collision — totally fine. HashMap stores them in the same bucket and uses equals to distinguish.
What's the default hashCode?
Object's default returns a value derived from the memory address — unique per object, not based on content.

Immutability & the final keyword

The three faces of final

final int x = 10;                 // variable: can't reassign
public final class Util { ... }   // class: can't be extended
public final void log() { ... }   // method: can't be overridden

Designing an immutable class

  1. Make the class final (so subclasses can't break immutability).
  2. Make all fields private final.
  3. No setters.
  4. Initialize all fields in the constructor.
  5. If fields reference mutable objects (Date, List), defensively copy in the constructor and in any getter.
public final class Money {
    private final long cents;
    private final String currency;

    public Money(long cents, String currency) {
        this.cents = cents;
        this.currency = currency;
    }

    public long getCents() { return cents; }
    public String getCurrency() { return currency; }

    public Money add(Money other) {
        return new Money(this.cents + other.cents, currency);   // new instance
    }
}

Why immutability matters

  • Thread-safe by default — no synchronization needed.
  • Safe to share / cache — like String pool.
  • Reliable as HashMap keys — hashCode can't change.
  • Easier to reason about — no surprises mid-execution.
Common Interview Q
Is String immutable in Java?
Yes. The class is final, all fields are final, no method modifies internal state.
final vs immutable?
final on a reference means you can't reassign the variable. The object itself can still be mutable. final List<Integer> nums = ...; — you can still call nums.add(5).

Threads & Runnable

A thread is a separate path of execution. The JVM lets you run multiple threads concurrently — for parallelism on multi-core CPUs, for handling I/O without blocking the UI, for processing requests in a server.

Two ways to create a thread

// 1. Extend Thread (less common)
class Worker extends Thread {
    @Override
    public void run() { System.out.println("Working..."); }
}
new Worker().start();

// 2. Implement Runnable (preferred — keeps inheritance free)
Runnable task = () -> System.out.println("Working...");
new Thread(task).start();
start() vs run()

Call start() to actually create a new thread. Calling run() directly just executes the code on the current thread — no parallelism.

Thread lifecycle

NEW ──start()──► RUNNABLE ◄────► RUNNING │ │ │ ▼ ▼ ┌─────────────┐ TIMED_WAITING │ BLOCKED │ (waiting for lock) (sleep, join, │ WAITING │ (wait/notify) wait(ms)) └─────────────┘ │ ▼ TERMINATED

Common thread methods

Thread.sleep(1000);              // pause current thread for 1 sec
t.join();                         // wait for thread t to finish
t.interrupt();                    // signal to stop
Thread.currentThread().getName(); // who am I?
t.setDaemon(true);                // dies when main thread dies
Common Interview Q
Difference between Thread and Runnable?
Thread is a class to extend; Runnable is an interface to implement. Runnable is preferred — Java has single inheritance, and Runnable lets you still extend something else.
What is a daemon thread?
A background thread that doesn't prevent JVM shutdown. When all non-daemon threads finish, JVM exits even if daemons are still running. Used for GC, monitoring, etc.

Synchronization

When multiple threads access shared mutable state, you get race conditions. Java offers several tools to make access safe.

The race condition

class Counter {
    int count = 0;
    void increment() { count++; }    // NOT atomic — read, add, write
}

// Two threads doing 1000 increments each — final count may be < 2000

synchronized

// On a method
public synchronized void increment() { count++; }

// On a block — control which lock you use
private final Object lock = new Object();
public void increment() {
    synchronized (lock) {
        count++;
    }
}

Synchronized acquires the object's intrinsic lock (monitor). Only one thread can hold the lock at a time.

volatile

Guarantees that reads/writes of a variable go directly to main memory — not cached in CPU registers. Good for flags read by multiple threads. Doesn't make compound operations (like x++) atomic!

private volatile boolean running = true;

public void stop() { running = false; }
public void run() {
    while (running) { ... }
}

Atomic classes

Lock-free atomic operations using CPU CAS (compare-and-swap) instructions.

AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();        // atomic ++
count.addAndGet(5);
count.compareAndSet(10, 20);

AtomicReference<String> name = new AtomicReference<>("Alice");

Explicit locks — ReentrantLock

private final Lock lock = new ReentrantLock();

public void doWork() {
    lock.lock();
    try {
        // critical section
    } finally {
        lock.unlock();             // always release in finally
    }
}

More flexible than synchronized — supports tryLock(), timed locks, fair ordering, multiple condition variables.

wait / notify / notifyAll

Low-level inter-thread communication. Must be called inside a synchronized block on the same object.

synchronized (queue) {
    while (queue.isEmpty()) queue.wait();  // release lock, wait
    process(queue.remove());
    queue.notifyAll();                      // wake up others
}
Common Interview Q
volatile vs synchronized?
volatile ensures visibility (reads see latest writes from other threads). synchronized ensures both visibility AND atomicity (locking). Use volatile for simple flags; synchronized when multiple operations must happen together.
What is a deadlock?
Two threads each hold a lock the other needs — both wait forever. Avoid by acquiring locks in a consistent order, using tryLock with timeouts.
synchronized vs ReentrantLock?
synchronized is simpler, automatic release on exit/exception. ReentrantLock is more flexible — interruptible, timed, fair, multiple conditions — but you must manually unlock.

Executors & CompletableFuture

Manually creating threads doesn't scale. The Executor framework manages a pool of reusable threads — you submit tasks, it schedules them.

ExecutorService basics

ExecutorService executor = Executors.newFixedThreadPool(4);

// Runnable — fire and forget
executor.execute(() -> System.out.println("Task done"));

// Callable — returns a value
Future<Integer> future = executor.submit(() -> {
    Thread.sleep(1000);
    return 42;
});

Integer result = future.get();   // blocks until done

executor.shutdown();             // important — releases threads

CompletableFuture — modern async

Java 8's powerful tool for chaining async operations.

CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> fetchUser(id))           // async step 1
    .thenApply(user -> user.getEmail())         // transform
    .thenCompose(email -> sendEmailAsync(email))// chain another async
    .thenAccept(System.out::println)            // side effect
    .exceptionally(ex -> {                      // error handling
        log.error("failed", ex);
        return null;
    });

// Combine multiple
CompletableFuture<String> combined = future1
    .thenCombine(future2, (a, b) -> a + b);

// Wait for all / any
CompletableFuture.allOf(f1, f2, f3).join();
CompletableFuture.anyOf(f1, f2, f3).join();

Virtual Threads (Java 21+)

Lightweight threads managed by the JVM, not the OS. You can have millions instead of thousands. Game-changer for I/O-bound workloads.

// Old: limited by OS thread count
Thread.ofPlatform().start(() -> doWork());

// New: virtually unlimited
Thread.ofVirtual().start(() -> doWork());

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i ->
        executor.submit(() -> handleRequest(i)));
}
Common Interview Q
Why use ExecutorService over new Thread()?
Thread creation is expensive. Pools reuse threads, limit total count, and provide proper task queuing, cancellation, and result tracking.
Future vs CompletableFuture?
Future is blocking — you call get() and wait. CompletableFuture is non-blocking — you compose callbacks with thenApply, thenCompose, etc.
When are virtual threads NOT a win?
CPU-bound work — virtual threads don't add parallelism beyond the carrier threads. They shine when threads spend most time blocked on I/O.

File I/O

Java has two APIs for file I/O: legacy java.io and modern java.nio.file. Use NIO whenever possible.

Reading a file (modern)

// All at once — small files
String content = Files.readString(Path.of("data.txt"));

// Line by line — any size, memory efficient
try (Stream<String> lines = Files.lines(Path.of("big.txt"))) {
    lines.filter(line -> line.contains("ERROR"))
         .forEach(System.out::println);
}

// As a list
List<String> lines = Files.readAllLines(Path.of("data.txt"));

Writing a file

Files.writeString(Path.of("out.txt"), "Hello, file!");
Files.write(Path.of("lines.txt"), List.of("line1", "line2"));

Legacy I/O with BufferedReader

try (BufferedReader br = new BufferedReader(new FileReader("data.txt"))) {
    String line;
    while ((line = br.readLine()) != null) {
        System.out.println(line);
    }
}

File operations

Path p = Path.of("docs/report.pdf");
Files.exists(p);
Files.size(p);
Files.copy(src, dst);
Files.move(src, dst);
Files.delete(p);
Files.createDirectories(Path.of("a/b/c"));
Files.walk(p).forEach(System.out::println);
Common Interview Q
Why use BufferedReader instead of FileReader directly?
FileReader reads one char at a time → many slow disk reads. BufferedReader reads big chunks at once → faster.
try-with-resources — why?
Automatically closes the resource on exit (normal or exception). Avoids forgetting finally { close(); }. Works for any AutoCloseable.

Interview Quick Reference

The 30 most common Java interview questions

Core Language
JDK vs JRE vs JVM?
JDK = JRE + dev tools (javac). JRE = JVM + libraries. JVM runs bytecode.
Is Java pass-by-value or by-reference?
Always pass-by-value. For objects, the value passed IS the reference (copy of address).
Why is String immutable?
Thread safety, String pool sharing, safe HashMap keys.
== vs equals?
== compares references; .equals() compares content.
final vs finally vs finalize?
final = constant/no-extend/no-override; finally = always-run block; finalize = deprecated cleanup method.
OOP
4 pillars of OOP?
Encapsulation, Inheritance, Polymorphism, Abstraction.
Abstract class vs Interface?
Abstract has state and constructors; interface is a contract with default/static methods (Java 8+). Java has single class inheritance but multiple interface inheritance.
Method overloading vs overriding?
Overloading = same name different params, compile-time. Overriding = subclass redefines parent method, runtime.
Can constructors be inherited?
No. But subclass constructors must call a parent constructor.
Can static methods be overridden?
No, only hidden. They're resolved at compile time.
Collections
ArrayList vs LinkedList?
ArrayList: array-backed, fast random access. LinkedList: node-backed, fast head/tail insertion, poor random access.
HashMap internal working?
Array of buckets + hash function. Collisions handled via linked list (or red-black tree for 8+ entries since Java 8). Resizes at load factor 0.75.
HashMap vs ConcurrentHashMap vs Hashtable?
HashMap: unsynchronized, fastest. Hashtable: legacy, fully synchronized, slow. ConcurrentHashMap: thread-safe, segment/bucket-level locking, much faster than Hashtable.
Why hashCode and equals must agree?
Equal objects must have equal hashCodes — otherwise HashMaps put them in different buckets and look-up fails.
fail-fast vs fail-safe iterators?
fail-fast: throws ConcurrentModificationException if collection modified during iteration (most non-concurrent collections). fail-safe: works on a copy or snapshot (CopyOnWriteArrayList, ConcurrentHashMap).
Exception & Memory
Checked vs unchecked exceptions?
Checked: must be declared/caught (IOException). Unchecked: extend RuntimeException, no compile enforcement (NullPointerException).
try-with-resources?
Auto-closes any AutoCloseable declared in the try() parentheses, even on exception.
Stack vs Heap memory?
Stack: per-thread, holds locals/refs/frames, fast, fixed size. Heap: shared, holds all objects, GC-managed.
Memory leak in Java?
References held when no longer needed — static collections, listeners not unregistered, ThreadLocals not cleared. GC can't reclaim them.
What is JIT?
Just-In-Time compiler — JVM dynamically compiles hot bytecode paths to native instructions for speed.
Multithreading
Thread vs Runnable?
Runnable preferred — keeps single inheritance free, separates task from execution mechanism.
synchronized vs volatile?
synchronized: lock-based, ensures atomicity + visibility. volatile: visibility only, no atomicity, no locking.
What is a deadlock?
Two threads each hold a lock the other needs. Prevent by consistent lock ordering.
Race condition?
Two threads access shared state without proper synchronization; outcome depends on timing.
Future vs CompletableFuture?
Future is blocking; CompletableFuture supports non-blocking composition (then-Apply, thenCombine, exceptionally).
Modern Java
What's a functional interface?
Interface with exactly one abstract method — can be implemented by a lambda.
map vs flatMap in Streams?
map: 1-to-1 transform. flatMap: 1-to-many, then flattens nested streams.
Why Optional?
Replaces nullable returns with explicit "value might be missing" type — forces handling.
What's a record?
Java 14+ compact syntax for immutable data classes — auto-generates constructor, accessors, equals, hashCode, toString.
Virtual threads?
Java 21+ lightweight threads — JVM-managed. Can have millions for I/O-bound concurrency.
