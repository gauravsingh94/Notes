What is SQL

SQL (Structured Query Language) is the language used to talk to relational databases. You ask the database for what you want — it figures out how to get it. That's the magic of SQL: it's declarative, not imperative.

A relational database stores data in tables — like spreadsheets — with rows and columns. Tables can reference each other through shared values (keys). Examples: MySQL, PostgreSQL, Oracle, SQL Server, SQLite.

Why SQL is everywhere

  • 50+ years old, still the dominant data language
  • Powers almost every business application (banking, e-commerce, CRM)
  • Same query syntax works across all major databases (95% portable)
  • Data analysts, backend devs, data engineers all need it

SQL vs NoSQL — when to use what

SQL: structured data, complex relationships, strong consistency, financial transactions. NoSQL: flexible schema, massive scale, simple lookups, document storage. Most interviews want you to know SQL inside out — it's the foundation.

Money rule

Never store money as FLOAT or DOUBLE. Floating-point math is approximate — you'll lose cents over millions of transactions. Always use DECIMAL.

SELECT Basics

SELECT is the workhorse of SQL — it reads data from tables. Let's start with a simple table.

3CarolSales75000

-- All columns, all rows
SELECT * FROM employees;

-- Specific columns
SELECT name, salary FROM employees;

-- Aliases (rename in output)
SELECT name AS employee_name, salary AS pay
FROM employees;

DISTINCT — remove duplicates

SELECT DISTINCT department FROM employees;
-- Engineering, Sales, Marketing

ORDER BY — sorting

SELECT name, salary FROM employees
ORDER BY salary DESC;            -- highest first

-- Multiple columns
SELECT name, department, salary FROM employees
ORDER BY department ASC, salary DESC;

LIMIT / OFFSET — pagination

-- Top 3 highest paid
SELECT name, salary FROM employees
ORDER BY salary DESC
LIMIT 3;

-- Page 2 of results (5 per page)
SELECT * FROM employees
ORDER BY id
LIMIT 5 OFFSET 5;
SQL syntax order vs execution order

You write: SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY … LIMIT …
The DB executes: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. That's why you can't use a column alias from SELECT in WHERE — it doesn't exist yet.

WHERE & Operators

WHERE filters rows. Only rows matching the condition are returned.

SELECT * FROM employees WHERE salary > 80000;
SELECT * FROM employees WHERE department = 'Engineering';
SELECT * FROM employees WHERE salary BETWEEN 70000 AND 85000;

Comparison operators

OperatorMeaning
=Equal

BETWEEN x AND yInclusive range

LIKE 'pattern'Pattern match
IS NULL / IS NOT NULLNull check

Logical operators

-- AND, OR, NOT
SELECT * FROM employees
WHERE department = 'Engineering'
  AND salary > 80000;

SELECT * FROM employees
WHERE department = 'Sales' OR department = 'Marketing';

-- Same as above but cleaner:
SELECT * FROM employees
WHERE department IN ('Sales', 'Marketing');

SELECT * FROM employees
WHERE NOT department = 'Engineering';
AND vs OR precedence

AND has higher precedence than OR. a OR b AND c is parsed as a OR (b AND c). When in doubt, parenthesize: (a OR b) AND c.

Common Interview Q
Why doesn't WHERE column = NULL work?
NULL represents "unknown" — and unknown = anything (even unknown) is also unknown. Use IS NULL instead.
Difference between BETWEEN and using >= AND <=?
None functionally — BETWEEN is shorthand. Both are inclusive of the endpoints.

INSERT, UPDATE, DELETE

INSERT — add rows

-- Single row
INSERT INTO employees (name, department, salary)
VALUES ('Eve', 'Engineering', 95000);

-- Multiple rows
INSERT INTO employees (name, department, salary)
VALUES
    ('Frank', 'Sales', 65000),
    ('Grace', 'Marketing', 72000);

-- Insert from a query
INSERT INTO archive_employees
SELECT * FROM employees WHERE department = 'Sales';

UPDATE — modify rows

-- Give Alice a raise
UPDATE employees
SET salary = 100000
WHERE name = 'Alice';

-- Multiple columns
UPDATE employees
SET salary = salary * 1.1,
    department = 'Senior Engineering'
WHERE department = 'Engineering' AND salary > 85000;
UPDATE without WHERE = disaster

UPDATE employees SET salary = 0 — without a WHERE — sets EVERY salary to 0. Always preview your filter with a SELECT first. Some DBs have a "safe mode" that blocks UPDATE/DELETE without WHERE.

DELETE — remove rows

-- Specific rows
DELETE FROM employees WHERE department = 'Marketing';

-- All rows (table structure remains)
DELETE FROM employees;

DELETE vs TRUNCATE vs DROP

CommandWhat it doesSpeedRollback
DELETE FROM tRemoves rows (logs each)SlowYes

Common Interview Q
DELETE vs TRUNCATE vs DROP?
DELETE removes rows (can have WHERE, slow, logged, rollback possible). TRUNCATE removes all rows fast (no WHERE, minimal logging). DROP removes the entire table including its definition.

Pattern Matching — LIKE, IN, BETWEEN

LIKE — string patterns

-- Names starting with 'A'
SELECT * FROM employees WHERE name LIKE 'A%';

-- Names ending with 'e'
SELECT * FROM employees WHERE name LIKE '%e';

-- Names with 'li' anywhere
SELECT * FROM employees WHERE name LIKE '%li%';

-- 4-letter names starting with 'A'
SELECT * FROM employees WHERE name LIKE 'A___';

Wildcards

WildcardMatches
%Any sequence of characters (including zero)
_Exactly one character

Case sensitivity depends on the DB. PostgreSQL is case-sensitive; MySQL is case-insensitive by default. Use ILIKE in PostgreSQL for case-insensitive, or wrap with LOWER().

IN — match against a list

SELECT * FROM employees
WHERE department IN ('Engineering', 'Sales', 'Marketing');

-- IN with a subquery
SELECT * FROM employees
WHERE department IN (
    SELECT name FROM departments WHERE budget > 1000000
);

BETWEEN — inclusive range

SELECT * FROM employees WHERE salary BETWEEN 70000 AND 90000;
SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';

Regular expressions (DB-specific)

-- MySQL / PostgreSQL
SELECT * FROM users WHERE email REGEXP '@gmail\\.com$';
SELECT * FROM users WHERE email ~ '@gmail\\.com$';     -- PostgreSQL

NULL Handling

NULL means "unknown" — not zero, not empty string. NULL behavior trips up everyone.

The NULL rules

NULL = NULL          -- NULL (not true!)
NULL != NULL         -- NULL
NULL + 5             -- NULL
NULL OR true         -- true
NULL AND true        -- NULL
NULL AND false       -- false  (because false stays false)

Testing for NULL

SELECT * FROM users WHERE phone IS NULL;
SELECT * FROM users WHERE phone IS NOT NULL;

-- WRONG — returns no rows
SELECT * FROM users WHERE phone = NULL;

NULL helper functions

FunctionWhat it doesExample
COALESCE(a, b, c)Returns first non-NULL valueCOALESCE(nickname, name, 'Anon')
IFNULL(a, b)If a is NULL, return b (MySQL)IFNULL(bonus, 0)
NVL(a, b)Same as IFNULL (Oracle)NVL(commission, 0)
NULLIF(a, b)NULL if a = b, else aNULLIF(price, 0)
ISNULL(x)Boolean test (some DBs)Use IS NULL instead
-- Replace NULLs with a default
SELECT name, COALESCE(phone, 'N/A') AS contact
FROM users;

-- Sum salaries including those with NULL bonus
SELECT name, salary + COALESCE(bonus, 0) AS total_pay
FROM employees;
NULL in aggregates

Aggregate functions (SUM, AVG, COUNT) ignore NULL — except COUNT(*) which counts every row. So COUNT(column) can differ from COUNT(*) when there are NULLs.

Common Interview Q
Difference between NULL and empty string?
NULL = unknown/missing; empty string = a known value that happens to be ""(zero length). '' IS NOT NULL is true.
Why AVG can give different results when there are NULLs?
AVG ignores NULLs entirely — divides sum by count of non-NULL values. If you want NULLs treated as 0, use AVG(COALESCE(col, 0)).

GROUP BY & HAVING

GROUP BY groups rows by one or more columns, then applies aggregates per group.

SELECT department,
       COUNT(*) AS headcount,
       AVG(salary) AS avg_salary,
       MAX(salary) AS top_salary
FROM employees
GROUP BY department;
departmentheadcountavg_salarytop_salary
Engineering28750090000
Sales17500075000
Marketing17000070000

The GROUP BY rule

Every non-aggregated column in SELECT must appear in GROUP BY. This is invalid:

-- ERROR — name is neither in GROUP BY nor an aggregate
SELECT department, name, COUNT(*)
FROM employees
GROUP BY department;

HAVING — filter on aggregates

WHERE filters rows before grouping. HAVING filters groups after.

-- Departments with more than 1 employee earning > 60k
SELECT department, COUNT(*) AS headcount
FROM employees
WHERE salary > 60000
GROUP BY department
HAVING COUNT(*) > 1;
WHEREHAVING
FiltersRowsGroups
RunsBefore GROUP BYAfter GROUP BY
Can use aggregates?NoYes
Common Interview Q
WHERE vs HAVING?
WHERE filters individual rows before they're grouped. HAVING filters groups after aggregation. WHERE can't use aggregates like SUM/AVG; HAVING can.
Can you use a column alias from SELECT in HAVING?
Depends on DB. MySQL/PostgreSQL: yes. Standard SQL: no — repeat the expression. Safer to repeat.

Mental model

INNER JOIN = intersection. Imagine two circles in a Venn diagram — INNER JOIN returns only the overlapping middle.

SELECT e.name, d.name AS department
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;

Result includes Dan (no department) AND Marketing (no employees).

Simulate it with: SELECT … LEFT JOIN … UNION SELECT … RIGHT JOIN …

Common Interview Q
Difference between INNER and LEFT JOIN?
INNER returns only matching rows. LEFT returns all rows from left table, with NULLs where the right has no match.
How to find rows in A but NOT in B?
LEFT JOIN B ON … then WHERE B.id IS NULL. Or use NOT EXISTS.

CROSS JOIN — Cartesian product

Every row of A combined with every row of B. If A has 3 rows and B has 4, you get 12 rows.

SELECT s.size, c.color
FROM sizes s
CROSS JOIN colors c;

Useful for generating all combinations (sizes × colors for a product catalog).

Accidental cross join

Forgetting the ON clause turns INNER JOIN into a cross join: SELECT * FROM a, b with no WHERE pairs every row of a with every row of b. Watch out for explosions on big tables.

Subqueries

A subquery is a query inside another query. Three flavors based on where you use it:

1. Scalar subquery — returns a single value

-- Employees earning more than the average
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

2. Row / Column subquery — returns a list

-- Employees in big-budget departments
SELECT name
FROM employees
WHERE dept_id IN (
    SELECT id FROM departments WHERE budget > 1000000
);

3. Correlated subquery — references outer query

Re-runs for each row in the outer query. Powerful but can be slow.

-- Employees earning more than their department's average
SELECT name, dept_id, salary
FROM employees e1
WHERE salary > (
    SELECT AVG(salary)
    FROM employees e2
    WHERE e2.dept_id = e1.dept_id        -- correlation!
);

Subquery in FROM (derived table)

SELECT dept_name, avg_salary
FROM (
    SELECT d.name AS dept_name, AVG(e.salary) AS avg_salary
    FROM employees e JOIN departments d ON e.dept_id = d.id
    GROUP BY d.name
) AS dept_stats
WHERE avg_salary > 75000;
Common Interview Q
Subquery vs JOIN — which is faster?
Usually JOIN. Most query optimizers can rewrite IN-subqueries as joins, but explicit JOINs are clearer and often faster — especially for correlated subqueries. Profile, don't assume.
When MUST you use a subquery?
When you need aggregation before filtering ("rows where x > AVG of all x"), or when comparing multiple aggregates per row.

EXISTS, ANY, ALL

EXISTS — does any row match?

Returns true as soon as one matching row is found — efficient for existence checks.

-- Customers who have placed at least one order
SELECT name FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

-- Customers who have NEVER placed an order
SELECT name FROM customers c
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.id
);

ANY / SOME — match any value in a list

-- Products costing more than ANY product in 'Books' category
SELECT name, price FROM products
WHERE price > ANY (
    SELECT price FROM products WHERE category = 'Books'
);
-- Means: greater than the MINIMUM book price

ALL — match every value

-- Products costing more than EVERY book
SELECT name, price FROM products
WHERE price > ALL (
    SELECT price FROM products WHERE category = 'Books'
);
-- Means: greater than the MAXIMUM book price
OperatorEquivalent
> ANY (set)> MIN(set)
> ALL (set)> MAX(set)
< ANY (set)< MAX(set)
< ALL (set)< MIN(set)
= ANY (set)IN (set)
Common Interview Q
EXISTS vs IN — when to use which?
EXISTS is usually faster — it stops at the first match. Prefer EXISTS for correlated subqueries on large tables. IN is cleaner for static lists.
What does SELECT 1 in an EXISTS clause mean?
Nothing special — EXISTS only checks if rows exist, not their content. SELECT 1, SELECT *, or any column works identically. Convention is SELECT 1 for clarity.

UNION, INTERSECT, EXCEPT

Set operations stack queries vertically — combining rows from multiple SELECTs.

UNION — combine, remove duplicates

SELECT name FROM customers
UNION
SELECT name FROM suppliers;

UNION ALL — combine, keep duplicates

Faster than UNION because no deduplication is needed.

SELECT customer_id, 'order' AS source FROM orders
UNION ALL
SELECT customer_id, 'return' AS source FROM returns;

INTERSECT — rows in both

-- Users who are both customers and suppliers
SELECT name FROM customers
INTERSECT
SELECT name FROM suppliers;

EXCEPT (MINUS in Oracle) — rows in first but not second

-- Customers who are NOT suppliers
SELECT name FROM customers
EXCEPT
SELECT name FROM suppliers;

Rules for set operations

  • Same number of columns in each SELECT.
  • Compatible data types in each position.
  • Column names taken from the first SELECT.
  • ORDER BY can only appear at the very end.
Common Interview Q
UNION vs UNION ALL?
UNION removes duplicates (requires sort/hash). UNION ALL keeps everything (much faster). Use UNION ALL unless you actually need deduplication.

CTEs (Common Table Expressions)

A CTE is a named temporary result set defined with WITH, used within a query. They make complex queries readable and reusable.

WITH high_earners AS (
    SELECT * FROM employees WHERE salary > 80000
),
by_dept AS (
    SELECT dept_id, COUNT(*) AS n, AVG(salary) AS avg_pay
    FROM high_earners
    GROUP BY dept_id
)
SELECT d.name, b.n, b.avg_pay
FROM by_dept b
JOIN departments d ON d.id = b.dept_id
ORDER BY b.avg_pay DESC;

Why CTEs beat subqueries

  • Read top-to-bottom like a story — vs. inside-out for nested subqueries
  • Can be referenced multiple times in the same query
  • Can be recursive (next section!)

Recursive CTE

The killer feature — generate sequences or traverse hierarchies.

-- Numbers 1 to 10
WITH RECURSIVE numbers AS (
    SELECT 1 AS n                       -- base case
    UNION ALL
    SELECT n + 1 FROM numbers
    WHERE n < 10                        -- termination!
)
SELECT * FROM numbers;
-- Walk an org chart: everyone under the CEO
WITH RECURSIVE org AS (
    SELECT id, name, manager_id, 1 AS depth
    FROM employees
    WHERE manager_id IS NULL            -- the CEO

    UNION ALL

    SELECT e.id, e.name, e.manager_id, o.depth + 1
    FROM employees e
    JOIN org o ON e.manager_id = o.id
)
SELECT * FROM org ORDER BY depth, name;
Recursive CTE = always have a termination condition

Forget the WHERE n < 10 and you've got an infinite loop. Most DBs cap recursion depth (default 100 in PostgreSQL) — but don't rely on that.

CASE WHEN

The SQL equivalent of if/else. Use it inside SELECT, ORDER BY, WHERE, or anywhere an expression is allowed.

SELECT name, salary,
    CASE
        WHEN salary > 90000 THEN 'High'
        WHEN salary > 70000 THEN 'Medium'
        ELSE 'Low'
    END AS salary_band
FROM employees;

Simple CASE

SELECT name,
    CASE department
        WHEN 'Eng'      THEN 'Tech'
        WHEN 'Sales'    THEN 'Business'
        ELSE 'Other'
    END AS division
FROM employees;

CASE in aggregates — the pivot trick

-- Headcount per department in ONE row (manual pivot)
SELECT
    SUM(CASE WHEN department = 'Engineering' THEN 1 ELSE 0 END) AS eng,
    SUM(CASE WHEN department = 'Sales'       THEN 1 ELSE 0 END) AS sales,
    SUM(CASE WHEN department = 'Marketing'   THEN 1 ELSE 0 END) AS mktg
FROM employees;

CASE in UPDATE

UPDATE employees
SET salary = CASE
    WHEN department = 'Engineering' THEN salary * 1.10
    WHEN department = 'Sales'       THEN salary * 1.05
    ELSE salary * 1.03
END;

Window Functions

Window functions are the superpower of modern SQL. They let you compute aggregates and rankings per row, while keeping the original rows visible.

The OVER clause

SELECT name, department, salary,
    AVG(salary) OVER ()                          AS overall_avg,
    AVG(salary) OVER (PARTITION BY department)   AS dept_avg
FROM employees;
namedepartmentsalaryoverall_avgdept_avg
AliceEngineering900008000087500
BobEngineering850008000087500
CarolSales750008000075000
DanMarketing700008000070000

The original 4 rows are kept — each is augmented with the relevant average.

Anatomy of OVER

function() OVER (
    PARTITION BY col1               -- like GROUP BY, but doesn't collapse rows
    ORDER BY col2                   -- order within each partition
    ROWS BETWEEN ... AND ...        -- which rows to include (the "frame")
)

Aggregate windows

-- Running total of sales by date
SELECT order_date, amount,
    SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

-- 7-day moving average
SELECT order_date, amount,
    AVG(amount) OVER (
        ORDER BY order_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS ma7
FROM orders;
Why windows are game-changing

Before windows, "the salary AND the department average on the same row" needed a self-join or subquery. Window functions do it in one elegant expression.

Ranking Functions

Three different ways to assign rank numbers.

SELECT name, salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num,
    RANK()       OVER (ORDER BY salary DESC) AS rnk,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS d_rnk
FROM employees;

Top N per group — the classic interview question

-- Top 3 highest-paid employees per department
WITH ranked AS (
    SELECT name, department, salary,
        DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rk
    FROM employees
)
SELECT name, department, salary
FROM ranked
WHERE rk <= 3;

NTILE — divide into N buckets

-- Salary quartiles
SELECT name, salary,
    NTILE(4) OVER (ORDER BY salary) AS quartile
FROM employees;

LAG, LEAD & Value Functions

These let you reach across rows in the same partition.

FunctionWhat it returns
LAG(col, n)Value from n rows before (previous)
LEAD(col, n)Value from n rows after (next)
FIRST_VALUE(col)First value in the window
LAST_VALUE(col)Last value in the window
NTH_VALUE(col, n)nth value in the window

Compare today vs yesterday

SELECT order_date, amount,
    LAG(amount) OVER (ORDER BY order_date) AS prev_amount,
    amount - LAG(amount) OVER (ORDER BY order_date) AS day_change
FROM daily_sales;

Difference from peak

SELECT name, salary, department,
    FIRST_VALUE(salary) OVER (
        PARTITION BY department
        ORDER BY salary DESC
    ) AS top_in_dept,
    FIRST_VALUE(salary) OVER (
        PARTITION BY department
        ORDER BY salary DESC
    ) - salary AS gap_from_top
FROM employees;
LAST_VALUE pitfall

Without a frame clause, LAST_VALUE defaults to "rows from start to current row" — so it returns the current row's value, not the last in partition. Fix: ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

CONCAT with NULL

In MySQL, CONCAT('hi', NULL, 'there') returns NULL. Use CONCAT_WS (with separator) which skips NULLs, or wrap fields with COALESCE.

Common Interview Q
Can a table have multiple primary keys?
No. But it can have a composite primary key — multiple columns combined into one key.
What's a surrogate key?
An artificial primary key (usually auto-incrementing integer or UUID) with no business meaning, vs a "natural key" like SSN or email which has real-world meaning. Surrogate keys are preferred — they never change.

Normalization

Normalization = organizing data to eliminate redundancy and avoid update anomalies. Each "normal form" adds stricter rules.

The denormalized horror

order_idcustomer_namecustomer_phoneproductsprices
1Alice555-1234Pen, Notebook5, 10
2Alice555-1234Pen5

Customer info repeated. Multiple values in one cell. Update Alice's phone → must update everywhere.

1NF — First Normal Form

Rule: Atomic values only. No repeating groups or lists in a cell.

Fix: Split "Pen, Notebook" into separate rows.

2NF — Second Normal Form

Rule: 1NF + every non-key column depends on the entire primary key (no partial dependencies). Matters when PK is composite.

3NF — Third Normal Form

Rule: 2NF + no transitive dependencies. Non-key columns must depend only on the primary key, not on other non-key columns.

Example violation: order(id, customer_id, customer_phone) — customer_phone depends on customer_id, not on order id. Fix: move it to a customers table.

BCNF — Boyce-Codd Normal Form

A stricter 3NF — every determinant must be a candidate key. Rarely matters in practice.

Properly normalized version

customers: id, name, phone
products: id, name, price
orders: id, customer_id, order_date
order_items: order_id, product_id, quantity

When NOT to normalize — denormalization

Sometimes you intentionally duplicate data for read performance — common in data warehouses and analytics tables. The trade-off: faster reads, harder writes (must update in multiple places).

Common Interview Q
Pros and cons of normalization?
Pros: less storage, no update anomalies, integrity. Cons: more joins (slower reads), harder to write some queries. Real systems balance — normalize transactional DBs, denormalize analytical DBs.
When is 3NF "enough"?
For 99% of business applications. BCNF/4NF/5NF are theoretical refinements; rarely needed in practice.

Indexes

An index is a separate data structure that lets the DB find rows faster — like a book's index page. Without it, every query has to scan the whole table.

CREATE INDEX idx_employees_email ON employees(email);
CREATE INDEX idx_employees_dept_salary ON employees(dept_id, salary);    -- composite
CREATE UNIQUE INDEX idx_employees_email ON employees(email);
DROP INDEX idx_employees_email;

How indexes work — B-tree

Most indexes are B-trees (balanced trees). Lookups in O(log n). The DB walks down the tree to find your value.

[Salary < 80k] / \ [50k, 60k, 70k] [80k, 90k, 100k+] / | \ / | \ rows rows rows rows rows rows

Index types

TypeUse case
B-treeDefault — equality, range, ORDER BY, prefix LIKE
HashEquality only (no range). Very fast for exact matches
BitmapLow-cardinality columns (gender, status). Warehouses
Full-textSearching inside large text (articles)
GIN / GiSTPostgreSQL: arrays, JSON, geo, full-text

Good for indexesBad for indexes

Composite indexes — leftmost prefix rule

Index on (a, b, c) can serve queries filtering by:

  • WHERE a = ? ✓
  • WHERE a = ? AND b = ? ✓
  • WHERE a = ? AND b = ? AND c = ? ✓
  • WHERE b = ? ✗ (no leftmost column)
  • WHERE a = ? AND c = ? ◇ (partial — only uses a)
Indexes aren't free

Every index slows down INSERT/UPDATE/DELETE — the DB must update each index too. They also take disk space. Don't index every column.

Common Interview Q
When does WHERE col LIKE '%abc' use an index?
Never (in B-tree). Leading wildcards skip indexes. LIKE 'abc%' CAN use one. Use full-text indexes for substring searches.
Does WHERE YEAR(date_col) = 2024 use an index on date_col?
No. Wrapping a column in a function defeats index use. Rewrite as WHERE date_col BETWEEN '2024-01-01' AND '2024-12-31'.

Views

A view is a saved SELECT query. You query it like a table, but it doesn't store data — it re-runs the query each time.

CREATE VIEW high_earners AS
SELECT id, name, department, salary
FROM employees
WHERE salary > 80000;

-- Now query it
SELECT * FROM high_earners WHERE department = 'Engineering';

-- Drop it
DROP VIEW high_earners;

Why views

  • Abstraction — hide complex joins behind a simple name.
  • Security — grant access to a view that exposes only safe columns.
  • Reusability — define once, use in many queries.
  • Consistency — everyone uses the same business logic.

Materialized views

A regular view re-runs every time. A materialized view stores the actual result and must be refreshed manually (or on a schedule). Faster reads, but data can be stale.

CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT DATE_TRUNC('month', order_date) AS month, SUM(amount) AS revenue
FROM orders
GROUP BY 1;

REFRESH MATERIALIZED VIEW monthly_revenue;
Common Interview Q
View vs Table?
Table holds data physically. View is a stored query that returns data on demand. Materialized view stores cached query results.
Can you INSERT into a view?
Sometimes — only "updatable views" with simple selects (no joins, no aggregates). Complex views are read-only.

Stored Procedures, Functions & Triggers

Stored procedures

A named, reusable block of SQL stored in the database. Called with CALL.

CREATE PROCEDURE raise_salary(IN dept_id INT, IN pct DECIMAL(5,2))
BEGIN
    UPDATE employees
    SET salary = salary * (1 + pct/100)
    WHERE dept_id = dept_id;
END;

CALL raise_salary(10, 5);     -- give Engineering a 5% raise

Functions

Like procedures but return a value. Can be used in queries.

CREATE FUNCTION yearly_salary(monthly DECIMAL)
RETURNS DECIMAL
DETERMINISTIC
BEGIN
    RETURN monthly * 12;
END;

SELECT name, yearly_salary(salary) FROM employees;

Triggers

Code that runs automatically in response to INSERT, UPDATE, or DELETE events.

CREATE TRIGGER audit_salary_change
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    IF OLD.salary != NEW.salary THEN
        INSERT INTO salary_audit (employee_id, old_salary, new_salary, changed_at)
        VALUES (OLD.id, OLD.salary, NEW.salary, NOW());
    END IF;
END;

Common Interview Q
When to use triggers?
Auditing, enforcing complex business rules, maintaining derived data. Use sparingly — triggers are invisible, hard to debug, and can chain unexpectedly.
Stored procedure vs application code?
Stored procs are faster (no network round trips) and DB-portable across apps. But they're harder to version-control, test, and scale than application code. Modern stacks favor application logic + simple SQL.

Transactions & ACID

A transaction is a group of SQL statements treated as a single unit. Either all of them succeed, or none of them — there's no in-between.

BEGIN;                                              -- START TRANSACTION

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT;     -- both succeed
-- or
ROLLBACK;   -- both undone

If the second UPDATE fails (or power goes out), ROLLBACK ensures account 1 isn't debited without account 2 being credited. No money disappears.

The ACID properties

LetterPropertyMeans
AAtomicityAll or nothing. Partial transactions don't exist.
CConsistencyTransactions move DB from one valid state to another (constraints respected).
IIsolationConcurrent transactions don't see each other's intermediate state.
DDurabilityOnce committed, changes survive crashes / power loss.

SAVEPOINT — partial rollback

BEGIN;
INSERT INTO orders ...;
SAVEPOINT after_order;

INSERT INTO order_items ...;
-- something fails
ROLLBACK TO after_order;     -- undo only the items, keep the order

COMMIT;
Classic example — bank transfer

Withdraw $100 from A, deposit $100 to B. Both must happen. If only the withdrawal succeeds, money vanishes. If only the deposit succeeds, money appears from nowhere. Transactions guarantee neither weird state can occur.

Isolation Levels

"Isolation" controls how much one transaction can see what others are doing. Stricter = safer but slower. Looser = faster but lets weird things happen.

The four read phenomena

ProblemWhat happens
Dirty ReadYou read data another transaction hasn't committed yet.
Non-repeatable ReadSame query in same transaction returns different values (someone updated between).
Phantom ReadSame query returns different rows (someone inserted between).
Lost UpdateTwo transactions update same row; one's change overwrites the other.

The four standard isolation levels

LevelDirtyNon-repPhantom
READ UNCOMMITTEDYesYesYes
READ COMMITTED (default in PG/Oracle)—YesYes
REPEATABLE READ (default in MySQL)——Yes*
SERIALIZABLE———

*MySQL's REPEATABLE READ also blocks phantom reads via "gap locks". Standard SQL allows them.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
-- ...
COMMIT;
Rule of thumb

Default isolation is fine for 99% of apps. Reach for SERIALIZABLE only when you really need it (banking, inventory). Higher isolation = more locks = lower throughput.

Common Interview Q
Difference between dirty read and non-repeatable read?
Dirty read = you see uncommitted data (might be rolled back). Non-repeatable read = you re-read and the value has changed (someone committed an update between).
Why isn't every database SERIALIZABLE by default?
Performance. SERIALIZABLE requires heavy locking or expensive validation. Most apps tolerate weaker isolation in exchange for higher throughput.

Locking & Deadlocks

Locks are how databases prevent two transactions from corrupting shared data. Most DB lock automatically — you usually don't see them, until they cause problems.

Lock types

LockAllows others to read?Allows others to write?
Shared (read lock)YesNo
Exclusive (write lock)NoNo

Lock granularity

  • Row-level — best concurrency, but more memory.
  • Page-level — locks blocks of disk.
  • Table-level — entire table locked. Cheap, but kills concurrency.

Explicit locking

-- Lock rows for update during transaction
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;

-- Lock rows for shared read
SELECT * FROM accounts WHERE id = 1 FOR SHARE;

Deadlock

Two transactions each waiting for a lock the other holds. Forever.

Transaction A Transaction B ───────────── ───────────── LOCK row 1 LOCK row 2 ◄ waits ► WAITING for row 2 WAITING for row 1 (held by B) (held by A)

Databases detect this and abort one transaction. You'll get an error and must retry.

Avoiding deadlocks

  • Acquire locks in a consistent order (e.g. always lower id first).
  • Keep transactions short — release locks fast.
  • Use lower isolation levels when possible.
  • Add retry logic in your application for the inevitable few.
Common Interview Q
Optimistic vs Pessimistic locking?
Pessimistic = acquire locks upfront (FOR UPDATE). Safer but slower under contention. Optimistic = no locks; check before commit (with a version column). Faster, but you must retry on conflict.

Query Optimization & EXPLAIN

The DB's query planner decides how to execute your query — which indexes to use, which join algorithm, what order to access tables. EXPLAIN shows you that plan.

EXPLAIN SELECT * FROM employees WHERE email = 'alice@x.com';

What EXPLAIN shows

  • Access method: table scan, index scan, index seek.
  • Join order: which table is read first.
  • Join algorithm: nested loop, hash, merge.
  • Estimated rows: how many rows the planner expects.
  • Cost: arbitrary internal units.

Common access methods

MethodSpeedWhen
Index SeekFastExact match on indexed column
Index ScanMediumRange query on indexed column
Table Scan / Full ScanSlowNo usable index — reads every row

Join algorithms

AlgorithmBest for
Nested LoopSmall outer + indexed inner table
Hash JoinNo useful index; both tables large; equi-join
Merge JoinBoth inputs already sorted on join key

Top 10 optimization tactics

  1. Add indexes on columns used in WHERE, JOIN, ORDER BY.
  2. Select only the columns you need — avoid SELECT *.
  3. Filter early — WHERE before JOIN when possible.
  4. Avoid functions on indexed columns in WHERE (kills index use).
  5. Use EXISTS instead of IN for correlated subqueries.
  6. Use UNION ALL when duplicates impossible — UNION is slower.
  7. LIMIT large result sets — don't fetch what you won't display.
  8. Watch out for N+1 query patterns in app code — join in SQL instead.
  9. Update statistics regularly (ANALYZE) so the planner has good info.
  10. Use covering indexes — index includes all columns needed → no table access.

The N+1 problem

-- BAD — 1 query for users + N queries for each user's orders
List<User> users = db.query("SELECT * FROM users");
for (User u : users) {
    u.orders = db.query("SELECT * FROM orders WHERE user_id = ?", u.id);
}

-- GOOD — single JOIN query
SELECT u.*, o.* FROM users u LEFT JOIN orders o ON o.user_id = u.id;
Common Interview Q
Why might EXPLAIN show a full scan even with an index?
Several reasons: small table (faster to scan), the function wraps the column, data types mismatch (implicit cast), low selectivity (90% of rows match), stale statistics.
What is a covering index?
An index that contains all columns needed by the query, so the DB doesn't need to fetch the table at all. Hugely faster.

SQL Interview Patterns

Certain query patterns appear over and over. Drill these and you'll recognize them in any interview.

1. Nth highest salary

-- Approach 1: window function
SELECT salary FROM (
    SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
    FROM employees
) t WHERE rk = N;

-- Approach 2: subquery
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);  -- 2nd highest

-- Approach 3: LIMIT
SELECT DISTINCT salary FROM employees
ORDER BY salary DESC LIMIT 1 OFFSET N-1;

2. Top N per group

-- Top 3 highest paid in each department
WITH ranked AS (
    SELECT *, DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
    FROM employees
)
SELECT * FROM ranked WHERE rk <= 3;

3. Find duplicates

SELECT email, COUNT(*)
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

4. Delete duplicates, keep one

-- Keep row with smallest id per email
DELETE FROM users
WHERE id NOT IN (
    SELECT MIN(id) FROM users GROUP BY email
);

5. Consecutive numbers / dates

-- Detect 3 consecutive identical values in a log
SELECT DISTINCT a.value
FROM logs a
JOIN logs b ON b.id = a.id + 1 AND b.value = a.value
JOIN logs c ON c.id = a.id + 2 AND c.value = a.value;

6. Running total

SELECT order_date, amount,
    SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

7. Median (no MEDIAN function in most DBs)

SELECT AVG(salary) AS median_salary FROM (
    SELECT salary,
        ROW_NUMBER() OVER (ORDER BY salary) AS rn,
        COUNT(*) OVER () AS total
    FROM employees
) t
WHERE rn IN ((total + 1) / 2, (total + 2) / 2);

8. Pivot — rows to columns

SELECT
    user_id,
    SUM(CASE WHEN month = 1 THEN amount ELSE 0 END) AS jan,
    SUM(CASE WHEN month = 2 THEN amount ELSE 0 END) AS feb,
    SUM(CASE WHEN month = 3 THEN amount ELSE 0 END) AS mar
FROM sales
GROUP BY user_id;

9. Month-over-month growth

WITH monthly AS (
    SELECT DATE_TRUNC('month', order_date) AS month,
           SUM(amount) AS revenue
    FROM orders GROUP BY 1
)
SELECT month, revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_month,
    (revenue - LAG(revenue) OVER (ORDER BY month)) * 100.0
        / LAG(revenue) OVER (ORDER BY month) AS pct_change
FROM monthly;

10. Gaps in sequence

-- Find missing IDs
SELECT id + 1 AS missing_start
FROM users u
WHERE NOT EXISTS (SELECT 1 FROM users WHERE id = u.id + 1)
ORDER BY id;

Top interview questions — quick answers

Top SQL Interview Q&A
DDL vs DML?
DDL defines schema (CREATE, ALTER, DROP). DML manipulates rows (INSERT, UPDATE, DELETE, SELECT).
DELETE vs TRUNCATE vs DROP?
DELETE removes rows (slow, logged, rollback OK). TRUNCATE removes all rows fast. DROP removes the table itself.
PRIMARY KEY vs UNIQUE?
PK: one per table, NOT NULL, identifies the row. UNIQUE: many allowed, NULLs allowed.
WHERE vs HAVING?
WHERE filters rows before grouping; HAVING filters groups after aggregation. HAVING can use aggregates.
INNER vs LEFT JOIN?
INNER returns matches only. LEFT returns all left rows, NULLs where no right match.
UNION vs UNION ALL?
UNION removes duplicates (slow). UNION ALL keeps everything (fast).
Why is NULL handling tricky?
NULL = unknown. NULL = NULL is NULL, not true. Use IS NULL / IS NOT NULL. Aggregates ignore NULL except COUNT(*).
Clustered vs Non-clustered index?
Clustered determines physical row order (one per table). Non-clustered is a separate structure pointing to rows (many allowed).
What is the ACID property?
Atomicity, Consistency, Isolation, Durability — the four guarantees of a transactional database.
What are the isolation levels?
READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE. Each prevents progressively more anomalies.
RANK vs DENSE_RANK vs ROW_NUMBER?
ROW_NUMBER: always unique. RANK: ties share rank, next is skipped (1,2,2,4). DENSE_RANK: ties share rank, no skip (1,2,2,3).
View vs Materialized View?
View re-runs every query. Materialized view stores results and must be refreshed; faster reads, possibly stale.
When does an index NOT help?
Leading wildcards in LIKE, functions wrapping the column, very small tables, low-selectivity columns.
Normalization vs Denormalization?
Normalization: reduce redundancy, enforce integrity (good for OLTP). Denormalization: duplicate data for read speed (good for OLAP / analytics).
How would you optimize a slow query?
EXPLAIN it → check for full scans → add appropriate indexes → reduce columns → push down filters → avoid functions on indexed cols → consider denormalization for hot paths.
